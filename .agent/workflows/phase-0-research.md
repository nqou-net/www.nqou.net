# Phase 0-R: テーマ調査（汎用）

## 概要
記事執筆に先立ち、テーマ（デザインパターン、アンチパターン等）についてウェブで最新情報を調査し、`agents/warehouse/` に保存する。
既存の調査情報がある場合は陳腐化の度合いを判定し、再利用または再調査を決定する。

## Step 1: 既存調査の確認と鮮度判定

`agents/warehouse/<slug>.md` が存在するか確認する。

### 1-1. 既存ファイルが存在する場合

フロントマターの `date`（調査実施日）と `staleness_category` を読み取り、以下の基準で再調査の要否を判定する。

| `staleness_category` | 説明 | 有効期間 | 例 |
|---|---|---|---|
| `stable` | 理論・原則。変化が極めて少ない | 18ヶ月 | GoFパターン定義、SOLID原則 |
| `moderate` | 実装慣習。ゆっくり変化 | 9ヶ月 | 言語固有の実装イディオム、アンチパターン |
| `volatile` | ツール・ライブラリ依存。頻繁に変化 | 3ヶ月 | フレームワーク固有の手法、ライブラリAPI |

**判定ロジック:**

```
経過日数 = 現在日 - date
有効期間を超過 → 再調査（Step 2 へ）
有効期間内 → 再利用（Step 3 へ）
```

`staleness_category` が未設定の場合は `moderate`（9ヶ月）をデフォルトとする。

### 1-2. 既存ファイルが存在しない場合

新規調査として Step 2 へ進む。

## Step 2: ウェブ調査の実行

`investigative-research` または `websearch-nerd` サブエージェントに調査を委任する。

### 2-1. サブエージェントへの指示テンプレート

```
以下のテーマについて、最新の技術情報をウェブで調査してください。

**テーマ**: <パターン名 / テーマ名>
**技術スタック**: Perl / Moo（該当する場合）
**調査観点**:
1. 公式・権威ある定義（GoF書籍、Refactoring Guru 等）
2. 最新の議論・批判・再評価（ブログ、カンファレンス発表等）
3. 類似パターンとの比較・使い分け
4. 実装上の注意点・よくある誤用
5. Perl/Moo での実装に関する知見（該当する場合）

**出力形式**: 以下の構造で Markdown を返してください。
- 各セクションに出典URL を明記
- 「事実」「推論」「仮定」を区別して記述
- 調査日を含める
```

### 2-2. 調査結果の保存

調査結果を `agents/warehouse/<slug>.md` に保存する。

**フロントマターの必須フィールド:**

```yaml
---
date: <調査実施日 ISO8601 +09:00>
description: '<テーマ>に関する調査結果'
staleness_category: <stable | moderate | volatile>
tags:
  - research
  - <テーマ関連タグ>
title: <テーマ>調査ドキュメント
---
```

**`staleness_category` の付与基準:**
- テーマが GoF パターン定義・SOLID 原則等の原理原則 → `stable`
- テーマが言語固有の実装手法・アンチパターン → `moderate`
- テーマがフレームワーク・ライブラリ・ツール固有 → `volatile`

### 2-3. 再調査の場合

既存ファイルを上書きする。旧版の内容は git 履歴で復元可能。

## Step 3: 調査結果の引き継ぎ

調査結果（新規・既存再利用・再調査いずれも）を Phase 0 の企画ステップに引き継ぐ。

Phase 0 では `agents/warehouse/<slug>.md` の内容をコンテキストとして読み込み、企画案の生成に活用する。

## 補足: 鮮度判定の override

ユーザーが明示的に「再調査して」と指示した場合は、有効期間内であっても再調査を実行する。
