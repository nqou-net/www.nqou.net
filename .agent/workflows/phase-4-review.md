# Phase 4: 記事レビュー・修正（汎用）

## 概要
Phase 3で執筆した記事に対して、一貫性、技術的正確性、指定されたルールの遵守状況をチェックし、問題があれば修正します。

## Step 1: レビュー基準の読み込み
対象の記事（`content/post/YYYY/MM/DD/<slug>.md`）と、使用した「シリーズマニフェスト」（および関連部品）を読み込みます。
可能であれば、同シリーズの直近3本程度の構造案または記事もあわせて確認し、既視感の強い反復がないか比較します。

## Step 2: 自己チェック（インスペクション）
以下の項目について、記事全体を厳しくチェックします。

### 1. ナラティブと視点 (Personas)
- [ ] **一人称の徹底**: `Personas` で定義された「語り部」の視点から一度も逸脱していないか？（例: 他人の内面を描写していないか）
- [ ] **キャラクターの言動**: 主人公や助手の言動が `Personas` のルール（口調、禁止事項、固定フレーズ）を守っているか？
- [ ] **語り部の固有性**: 語り部の話し方や反応が、その回のプロファイルに固有のものになっているか？ 毎回同じワトソン口調に潰れていないか？
- [ ] **固定フレーズの過剰反復なし**: 決め台詞、事務所描写、定番ツッコミを機械的に反復していないか？
- [ ] **導入の連鎖が定型化していない**: 「LCI名乗り → 雑居ビルの三階 → 扉を開けると」のような固定した入り方を、そのまま焼き直していないか？
- [ ] **語り部のツッコミ温度が使い回しでない**: 毎回同じ距離感の心中ツッコミや、同じ調子の困惑で処理していないか？
- [ ] **初対面の整合性**: 特殊回を除き、語り部がロックと初対面である前提を壊していないか？ 事前知識がある場合は、その情報源が本文中で明示されているか？
- [ ] **質問役として機能している**: 語り部が理解のための質問を実際に発し、技術説明が会話として展開されているか？ 地の文だけで賢く理解しすぎていないか？
- [ ] **呼称の距離感が自然**: 特殊な関係変化がない限り、地の文で主人公を「ロックさん」と呼ぶなど、初対面相応の距離感を保っているか？

### 2. 世界観と用語 (Metaphors & Plots)
- [ ] **メタファーの一貫性**: 見出しや本文中の比喩が `Metaphors` で定義された世界観に合致しているか？（探偵用語と医療用語が混ざるなどの破綻はないか）
- [ ] **プロットの進行**: `Plots` で定義された幕構成（5幕、4幕など）に沿って物語が進行しているか？
- [ ] **展開の既視感なし**: 直近回と比べて、導入、質問の運び、解決の見せ方、オチが焼き直しになっていないか？
- [ ] **Phase 1 の変化点を反映**: 構造案ファイルに記した「今回の差分」が本文に実装されているか？
- [ ] **診断シーンの運びが固定化していない**: テーマ比喩を一言で言い当ててから毎回同じリズムで犯人名を宣告していないか？
- [ ] **結末のギャグが固定化していない**: 「報酬は○○と同じ杯数」といった数合わせオチや、標語的な説教締めを惰性で繰り返していないか？
- [ ] **ワトソン君呼びの文脈が省略されていない**: 語り部がその呼称を既知のものとして扱っていないか？ 既知なら、その前提が本文で共有されているか？

### 3. 出力形式 (Formats)
- [ ] **フロントマター**: `Formats` で指定された必須タグやカテゴリが正しく設定されているか？
- [ ] **記事末尾のフォーマット**: 記事の最後に、指定された「まとめ（報告書、処方箋など）」が正しいフォーマットで配置されているか？

### 4. 技術の正確性 (Code)
- [ ] **コードと解説の整合性**: Phase 2で検証したコードが正しく引用されており、解説に技術的な誤りがないか？

## Step 3: 修正（リファクタリング）
上記のチェック項目で1つでも「No (未達)」がある場合、該当箇所を修正して記事ファイルを上書き保存します。
とくに「ルール違反ではないが、毎回同じに読める」という問題は見逃さず、語り部の声と場面運びを再設計して修正してください。
**すべてのチェック項目が「Yes」になるまで、Step 2とStep 3を繰り返してください。**

## Step 4: 完了報告
全てのチェックがパスしたら、レビュー完了の旨と最終的な記事ファイルのパスをユーザーに報告してワークフローを終了します。
