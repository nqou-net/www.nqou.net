# Article Style Check (記事スタイルチェック)

## 概要

記事のMarkdownファイルに対して、プロジェクトで繰り返し発生しているスタイル違反を検出し、必要に応じて修正するスキルです。

## 使用方法

ユーザーから以下の情報を受け取ってください。

1. **対象**: 以下のいずれか
   - ファイルパス（例: `content/post/2026/03/25/070705.md`）
   - glob パターン（例: `content/post/2026/03/**/*.md`）
   - 指定なし → `draft: true` の全記事を対象とする
2. **モード**:
   - **チェック**（デフォルト）: 違反の一覧を報告するのみ
   - **修正**（`--fix`）: 自動修正可能な項目を修正し、AI判断項目は報告のみ

---

## ルール定義

### 自動修正可能（検出・修正ともに機械的に実行可能）

| ID | ルール名 | 重要度 | 検出・修正方法 |
|---|---|---|---|
| S01 | 会話文中の太字 | High | `「」` を含む行の `**...**` を検出し、`**` を除去する |
| S02 | セクションラベルの太字 | High | `**【...】**` を検出し、`**` を除去して `【...】` にする |
| S03 | 箇条書き末尾の句点 | High | `^- ` で始まる行の末尾 `。` を除去する。ただし `。」` `。）` `。）` など引用・括弧内の句点は除外 |
| S05 | フロントマター必須フィールド | Medium | `title`, `date`, `tags`, `categories`, `description` の存在を確認する |
| S06 | タグ形式 | Medium | 各タグが `/^[a-z0-9]+(-[a-z0-9]+)*$/` に合致し、5個以内であることを検証する |
| S07 | コードブロック言語タグ | Medium | ` ``` ` の直後に言語指定がないものを検出する（警告のみ。言語が不明なため自動修正はしない） |
| S08 | H1使用禁止 | Medium | `^# ` で始まる行を検出する（`## ` への変更を提案） |

### AI判断が必要（検出は可能だが、修正に文脈理解が必要）

| ID | ルール名 | 重要度 | 検出方法と判断理由 |
|---|---|---|---|
| S04 | 箇条書きの文体 | High | `^- ` で始まる行のです・ます調の語尾（`です`, `ます`, `ません`, `でしょう`, `ください`）を検出する。だ・である調への書き換えは文意の保持が必要なためAI判断 |
| S09 | 内部リンク形式 | Low | `/content/post/YYYY/MM/DD/HHMMSS.md` パターンを検出し、パーマリンク形式 `/YYYY/MM/DD/HHMMSS/` への変換を提案する |
| S10 | ショートコード形式 | Low | 本文中のベアURL（`https://` で始まるリンク）がショートコード（`linkcard` / `amazon`）になっていないものを報告する |
| S11 | Mermaid図検証 | Low | `mermaid` コードブロックの構文妥当性（開始キーワード `classDiagram`, `sequenceDiagram`, `graph` 等の有無）を確認する |
| S12 | バックスラッシュエスケープ | Low | コードブロック内の `\\` が意図通りかを確認する（要確認フラグのみ） |

---

## 実行ステップ

### Step 1: 対象ファイルの特定

ユーザーの入力に基づいて対象ファイルを収集する。

- **ファイルパス指定**: そのファイルを対象とする
- **glob指定**: パターンに一致するファイルを対象とする
- **指定なし**: `content/post/` 配下で `draft: true` を含むファイルを `Grep` ツールで検索し、対象とする

### Step 2: コードブロック領域の識別

各ファイルを読み込み、` ``` ` で囲まれたコードブロック領域の行番号範囲を記録する。

- 最も外側の ` ``` ` ペアのみをコードブロックとして扱う
- S01, S02, S03, S04, S08 のチェックでは、この範囲内の行をスキップする
- S07 はコードブロックの開始行 ` ``` ` 自体を検査対象とする

### Step 3: フロントマター領域のチェック（S05, S06）

ファイル先頭の `---` で囲まれたYAMLフロントマターを解析する。

- S05: 必須フィールド（`title`, `date`, `tags`, `categories`, `description`）の存在を確認する
- S06: `tags` 配列の各要素が正規表現に合致し、要素数が5個以内であるかを検証する

### Step 4: 本文のチェック（S01〜S04, S07, S08〜S12）

コードブロック外の本文に対して、各ルールを適用する。

**自動修正可能ルール:**
- S01: `「` を含む行で `**テキスト**` → `テキスト` に置換する
- S02: `**【テキスト】**` → `【テキスト】` に置換する
- S03: `^- ` 行の末尾 `。` を削除する（引用・括弧内は除外）
- S07: ` ``` ` 直後に言語指定がないものを警告する
- S08: `^# ` で始まる行を警告する

**AI判断ルール:**
- S04: `^- ` 行のです・ます調を検出し、だ・である調への書き換え案を提示する
- S09: `/content/post/` を含むリンクを検出し、パーマリンク形式への変換を提案する
- S10: ベアURLを検出し、ショートコード化を提案する
- S11: Mermaid図の構文を確認する
- S12: コードブロック内のエスケープ状況を報告する

### Step 5: 結果報告

以下の形式で報告する。

```markdown
## チェック結果

### 違反一覧

| ファイル | 行 | ルール | 重要度 | 内容 | 修正案 |
|---|---|---|---|---|---|
| content/post/2026/03/25/070705.md | 42 | S01 | High | 会話文中の太字 | `**見事だ**` → `見事だ` |
| ... | ... | ... | ... | ... | ... |

### サマリー
- 対象ファイル数: N
- 違反総数: N（自動修正可能: N / 要判断: N）
```

### Step 6: 修正の実行（修正モード時のみ）

1. 自動修正可能な違反（S01, S02, S03, S08）を `Edit` ツールで適用する
2. 修正前後の diff を表示する
3. AI判断が必要な違反は報告のみとし、ユーザーに判断を委ねる
4. 修正後に `mise run test` を実行してビルドが通ることを確認する

---

## 品質ゲート

1. **コードブロック保護**: コードブロック（` ``` ` 〜 ` ``` `）内のテキストは絶対に改変しない
2. **フロントマター保護**: フロントマターの `---` 区切りを壊さない
3. **ビルド検証**: 修正後は必ず `mise run test` でビルドが通ることを確認する
4. **AI判断ルールの承認**: S04, S09〜S12 はユーザーの承認なしに自動修正しない
