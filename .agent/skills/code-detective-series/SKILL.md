# Code Detective Series

## 概要

「コード探偵ロックの事件簿」シリーズ記事を、段階的なワークフローで破綻なく生成する。
汎用ワークフロー（`series-unified-*`）をベースに、探偵シリーズ固有のキャラクター制約と物語ルールを注入する。

## 創作の大原則

> **ドラマに魅力があれば、あらゆる設定は柔軟に変えてよい。**

以下はすべて「出発点」であり「上限」ではない。物語をより面白くするためなら、積極的に逸脱・拡張すること。

- シーン配分は自由。現場検証が5%で推理披露が60%でもいい。逆もまた然り。
- ワトソン君は1人とは限らない。2人の依頼人が同時に来ても、ロックが出張捜査に行ってもいい。
- 時系列の入れ替え、叙述トリック、回想構成、手記形式など、物語構造は**なんでもあり**。
- ロックが珍しく負ける回、ワトソン君がロックを出し抜く回があってもいい。
- 唯一の不変ルール：**技術的に正確であること**と**読者が楽しめること**。

## 事前確認

1. ユーザー入力から **アンチパターン名** と **解決に使うデザインパターン名** を抽出する。
2. `slug` をケバブケースで決定する（例: `code-detective-strategy-god-class`）。
3. 既存成果物の有無を確認する。
   - `agents/structure/code-detective-*`
   - `agents/tests/code-detective-*`
   - `content/post/YYYY/MM/DD/*.md`

## キャラクター設定の読み込み

**すべてのフェーズの前に**、以下を読み込み参照すること：

- `.agent/components/personas/code-detective.md`（キャラクターの核・物語構造・ルール）

ただしこれらは「基本設定集」であって「制約の牢獄」ではない。ドラマのために設定を拡張・変形することを歓迎する。

## 実行チェイン

### Phase 0: 事件設計（探偵シリーズ固有）

ユーザーと対話しながら以下を決定する：

1. **アンチパターン（容疑者）の選定**: God Class、Spaghetti Code、Shotgun Surgery 等
2. **解決パターン（推理の切り札）の選定**: Strategy、Observer、Facade 等
3. **ワトソン君のプロファイル設計**: 性別、人数、職種、苦境の内容（毎回変わる。複数人でもいい）
4. **事件メタファーの作成**: タイトル命名規則を参考に、自由にアレンジ可
5. **物語のフック**: この回ならではの仕掛け（叙述トリック、時系列の入れ替え、ロックの敗北、等）

- **停止条件**: 上記の設定をユーザーに提示し、承認を得るまで Phase 1 に進めない。

### Phase 1: Prepare

- 実行: `/series-unified-prepare` ワークフロー
- 生成する: `agents/structure/code-detective-<slug>.md`（構造案ファイル）
- 構造案にはストーリー構成を含めるが、形式は物語に合わせて自由に設計する

### Phase 2: Code

- 実行: `/series-unified-code` ワークフロー
- 生成する: `agents/tests/code-detective-<slug>/lib/` 以下に Before/After コード
- Perl + Moo を使用（シリーズの統一言語）
- コード自体は技術的に正確であること（ここだけは妥協しない）

### Phase 3: Write

- 実行: `/series-unified-write` ワークフロー
- 読み込む:
  - `.agent/components/personas/code-detective.md`
  - 構造案ファイル
  - テストディレクトリのコード
- 生成する: `content/post/YYYY/MM/DD/NNNNNN.md`
- **守るべき核**:
  1. 地の文はワトソン君の視点（ただし複数ワトソンの場合は交替・並行もあり）
  2. ワトソン君は知識はあるがひらめきがない（ただし稀に逆転する回もあり得る）
  3. 記事末尾に「探偵の調査報告書」を配置
  4. **技術的に正確なコードと解説**
- **フロントマター規約**:
  - `categories: [tech]`（固定。`Design Patterns` や `Perl` をカテゴリに入れない）
  - 必須タグ: `design-pattern`, `perl`, `moo`, `{pattern名}`, `{アンチパターン名}`, `refactoring`, `code-detective`
  - アンチパターン名はケバブケースで（例: `god-class`, `shotgun-surgery`, `duplicated-code`）
  - 禁止: 日本語タグ、`oop` 等のシリーズ外タグ

### Phase 4: Visual

- 扉絵: `/code-detective-visual-header` ワークフロー
- 挿絵: `/series-unified-visual` ワークフロー
- スタイルは物語の雰囲気に合わせて自由に選択

### Phase 5: Review

- 実行: `/series-unified-review` ワークフロー
- **探偵固有の確認**（ワークフロー内「コード探偵シリーズの追加チェック」セクション参照）:
  - [ ] ワトソン君のリアクションが物語として自然か（「物分かりが良すぎる」問題がないか）
  - [ ] 読者が技術的にも物語的にも楽しめる記事になっているか
  - [ ] 「探偵の調査報告書」が配置されているか
  - [ ] コードが正確に動作するか

## 品質ゲート

1. Phase 0 承認前に Phase 1 へ進めない。
2. Phase 2 はテストを通す。コードの正確性だけは絶対に妥協しない。
3. Phase 3 は末尾に `## 探偵の調査報告書` を必ず付ける。
4. Phase 5 は全チェック項目が通るまで差し戻しと修正を繰り返す。
