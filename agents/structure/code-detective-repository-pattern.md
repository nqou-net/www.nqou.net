## 語り部プロファイル

- **名前**: 中原（なかはら）
- **一人称**: 僕
- **年齢**: 30歳
- **職種**: BtoB SaaS「OrderHub」（受発注管理プラットフォーム）のバックエンドエンジニア
- **経験**: 6年
- **性格**: 真面目で責任感が強いが、自分の書いたコードへの愛着が強く防衛的。「saveがモデルにあるのは普通だろう」と思い込んでいる。指摘されると最初は反発するが、論理的に示されると受け入れる。テストの重要性は理解しているが、「テストが遅い＝テスト環境の問題」と考えており、設計の問題だとは思っていない。
- **背景**: OrderHubの初期開発メンバー。注文管理の `Order` クラスを自分で書いた。3年間の機能追加でビジネスロジックとSQL（DBI直叩き）が同居するファットモデルに肥大化。テストスイートは毎回テスト用DBを立ち上げるため30分かかり、CIがタイムアウトで頻繁に落ちる。
- **話し方の輪郭**: 実務寄りの語彙。「移行コスト」「既存のテスト資産」など運用目線の言葉が出る。比喩はあまり使わない。反論するとき語気が少し硬くなる。
- **主人公との距離感**: 最初は「外部の人間に何がわかるのか」という警戒。シニアエンジニアからの紹介なので礼儀は保つが、内心で「本当に役立つのか」と疑っている。コードの問題点を的確に指摘されていくにつれ、防衛壁が崩れていく。
- **事前知識**: 社内のシニアエンジニア（高木さん）が「面白い奴がいる」とSlackで紹介。LCIの名前と「Perlに強いらしい」程度の情報。会うのは初めて。
- **呼称**: 地の文では「ロックさん」。ただし心中ツッコミでは「この男」「この人」。
- **ワトソン君呼びへの反応**: 1回目「中原です」と訂正。2回目は呆れて流す。3回目以降は無視して質問に移る。

## 変化点（v1からの差分）

1. **導入の差分**: 事務所訪問型 → **現場出張型**。中原のオフィスにロックが来る。雑居ビル描写なし。会議室でPCを広げる。
2. **語り部の声の差分**: 素直な受容型 → **防衛的・実務質問型**。「saveがモデルにあるのは普通だ」「移行コストはどうなる」と食い下がる。
3. **対話の差分**: 困惑→質問→即納得 → **反発→論破される→渋々認める→実装コストを確認→最後に自ら応用を考える**。理解が段階的に進む。
4. **終わり方の差分**: 報酬ギャグ＋標語 → **中原の独白型**。Slackに報告を書きながら、自分の言葉でリファクタリングの意味を噛み砕く。ロックの言葉の引用ではなく、自分自身の気づきとして締める。
5. **関係性の差分**: 初対面の困惑ではなく、紹介経由で「実力は知らないが、とりあえず話だけ聞こう」という半信半疑スタート。
6. **質問の差分**: 「でも〜ですよね？」型の受動的確認 → 「移行時に既存テストはどうなる？」「Repositoryのインターフェースが増えたらどうする？」など運用目線の攻めた質問。

## 今回あえて避ける定型

- **雑居ビル導入連鎖**: 「階段を上がる→LCI看板→ドアを開けると排熱・缶・本」を使わない
- **「初歩的なにおいだよ、ワトソン君」**: 開口一番の犯人宣告に使わない
- **物品要求報酬ギャグ**: キーボード、技術書、エナジードリンク杯数ネタを使わない
- **「～するな、ワトソン君」標語締め**: 末尾の標語型を使わない

## プロット（5幕構成）

### I. 導入（依頼）— 見出し「CIが赤く染まった日」
- 書き出し: 中原の独白で始まる。CIダッシュボードの赤いバッジ、3日間マージできないPR、チームの士気低下。
- 紹介経由: 高木さん（シニアエンジニア）がSlackで「テストの設計に詳しい人がいるから、一度話してみろ」と紹介。
- ロック登場: 会議室に現れる。自己紹介なし。中原のPCを覗き込み「Orderクラスを見せたまえ」。
- 中原の反応: 「いきなりコードか」と面食らうが、高木さんの紹介なので付き合う。
- ロックの第一声は「初歩的な〜」ではなく、コードを見てから「この容疑者は二重生活を送っているね」のような言い回し。

### II. 現場検証 — 見出し「二重生活を送る容疑者」
- Before コード提示: `Order` クラスの全体
- 中原が自分のコードを弁護: 「当時は小さかったんです。saveがモデルにあるのは自然でしょう」
- ロックが責務を分析: ホワイトボード（会議室の）に二分割して可視化
- 中原が反論: 「でも confirm の中で save を呼ぶのは、ステータスを変えたら即反映したいからです」
- ロックが論破: 矢印を引いて依存チェーンを可視化。「confirm をテストするには save が要る。save が動くには dbh が要る。dbh が要るということは DB が要る」
- Mermaid図: Fat Model の依存構造
- 中原渋々認める: 「確かに……total のテストを書くだけでも FakeDbh が要って……」

### III. 推理披露 — 見出し「供述を分離せよ」
- ロックが解決策を示す: 「容疑者のダイエットだ」ではなく、「この容疑者には弁護士を付ける。供述（データアクセス）は弁護士（Repository）を通してのみ行われる」
- After Order（純粋なドメインモデル）
- 中原の質問: 「保存や検索は誰がやるんですか」→ Repository ロール定義
- DbiOrderRepository 提示
- 中原の運用質問: 「テスト用にも DB 接続が要りますよね？」
- InMemoryOrderRepository 提示 — ここでロックが「これが核心だ」
- 利用コード（本番・テスト）
- Mermaid図: 分離後の構造
- 中原の攻めた質問: 「Repository のインターフェースが肥大化したら？」→ ロックが Query Object / Specification パターンに言及しつつ「まずは3メソッドで始めるべきだ」

### IV. 解決 — 見出し「30分が3秒になった日」
- テスト結果の提示
- 中原がテスト結果を見て声を上げる
- ロックは淡々と結果を眺める
- 報酬ギャグなし。代わりにロックが静かに去る準備をする。
- 中原が引き留めかけるが、ロックは振り返らずに一言残す（標語ではなく、含みのある言葉）
