## 語り部プロファイル

- **名前**: 中原（なかはら）
- **一人称**: 僕
- **年齢**: 30歳
- **職種**: BtoB SaaS「OrderHub」（受発注管理プラットフォーム）のバックエンドエンジニア
- **経験**: 6年
- **性格**: 真面目で責任感が強い。コードの品質を上げたいという意志はあるが、レガシーコードの壁に阻まれて手が出せないでいる。テストの重要性は理解しているが、現状のテストスイートの遅さに苛立ちを感じている。
- **背景**: OrderHubの初期開発メンバー。注文管理の `Order` クラスを自分で書いたが、機能追加を重ねるうちにビジネスロジックとSQL（DBI直叩き）が同居するファットモデルに肥大化。テストスイートは毎回テスト用DBを立ち上げるため30分かかり、CIがタイムアウトで頻繁に落ちる。「注文ステータスの遷移ロジックだけテストしたいのに、なぜDBが要るんだ」という苦しみを抱えている。

## プロット（5幕構成）

### I. 導入（依頼）— 見出し例: 「CIが止まった朝」
- 中原がLCIを訪問。「テストスイートが30分かかる。CIがタイムアウトで落ちて、PRがマージできない」
- 事務所描写（排熱、エナジードリンク缶）
- ロック「初歩的なにおいだよ、ワトソン君。モデルが太りすぎている」
- 中原の巻き込まれ描写（PCを取り上げられる等）

### II. 現場検証 — 見出し例: 「肥大化した容疑者の取り調べ」
- Before コード提示: `Order` クラスにビジネスロジック（ステータス遷移、合計金額計算）とデータアクセス（DBI直叩きの `save`, `find_by_id`, `search`）が混在
- ロックが問題を指摘: 「このモデルは知りすぎている。自分のビジネスルールだけでなく、どのテーブルに住んでいるか、どんなSQLで呼ばれるかまで知っている」
- アンチパターン「Fat Model」の特定: テストにDBが必須→遅い→書かない→品質低下の悪循環
- Mermaid図: Order クラスの責務が肥大化している様子

### III. 推理披露 — 見出し例: 「データの番人を分離せよ（Repository）」
- Repository ロール（インターフェース）の定義: `save`, `find_by_id`, `search`
- DbiOrderRepository（本番用）: DBI を使った実装
- InMemoryOrderRepository（テスト用）: ハッシュで保持するインメモリ実装
- Order クラスからデータアクセスを完全に除去→純粋なビジネスロジックだけに
- 利用コード: Repository を注入して使う
- Mermaid図: Order と Repository の分離

### IV. 解決 — 見出し例: 「30分が3秒に」
- テスト結果の提示
- Before: DB必須、テスト30分、ステータス遷移のテストにもDB接続
- After: InMemoryRepository でテスト3秒、ビジネスロジックだけを高速にテスト
- 報酬シーン（ヴィンテージ技術書 or 珍しいキーボード）
- 注意点: Repository は万能ではない。複雑なクエリやJOINが必要な場合は、Repository のインターフェースが肥大化する危険がある

### V. 報告書 — 「探偵の調査報告書」
- テーブル: Fat Model → Repository パターン
- 推理のステップ: 4段階
- ロックより: データアクセスとビジネスロジックの分離、テスタビリティの重要性
