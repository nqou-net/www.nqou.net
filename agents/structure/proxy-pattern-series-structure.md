# 連載構造案：Proxyパターンを学ぶ新シリーズ

調査結果: `content/warehouse/proxy-pattern.md` に基づく

## 前提情報

- **技術スタック**: Perl v5.36以降（signatures、postfix dereference対応）、Mooによるオブジェクト指向プログラミング
- **想定読者**: Perl入学式卒業程度の初学者
- **想定ペルソナ**: 「Mooで覚えるオブジェクト指向Perl」入門連載を修了し、もう一段レベルアップしたい人
- **学習目標**:
  - Proxyパターンが「アクセス制御・遅延初期化・キャッシュ」を自然に解決する理由を体験する
  - SOLID（特にSRP/OCP）とProxyの関係を体感する
  - Mooで「代理」オブジェクトを実装し、責務分離を自分の手で作れるようになる
- **位置づけ**: デザインパターン学習シリーズの構造パターン編（別テーマで刷新）
- **ストーリー**: 既知の知識で動くものを作る → 機能追加で破綻 → Proxy導入で解決 → 完成コードで自慢
- **制約**:
  - 1記事1概念
  - コード例2つまで
  - 回の最後には完成コードを提示（原則1ファイル）
  - シリーズ名にデザインパターン名は入れない
  - 教科書的な文法解説から入らない
  - ちょっと生意気・ハッキング的な遊び心を含める

## 案A: 「ゴーストギャラリー」アプローチ

### シリーズ名案

**「ゴーストギャラリー・ビューワを作ろう」**（全5回）

### 特徴・アプローチ

呪いの高解像度アートを「覗き見」できるギャラリーを作る。最初は画像を全部読み込んで固まる→遅延初期化→アクセス制御→キャッシュ→監査ログと段階的に「代理」役を増やし、Proxyパターンを一気に体感する。見た目が派手でデモに強いのが特徴。

### USP（独自の価値提案）

**「なぜこのシリーズにお金を払う価値があるのか？」**

- **“遅い/重い/危ない”を全部代理で片付ける感覚を、手を動かして体感できる**
- **ギャラリーという派手な題材で、完成物を見せて自慢できる**
- **「Proxy＝地味」という先入観を破壊し、面白さと実用性を両立する**

### メリット

- ビジュアルで成果がわかりやすい
- Virtual/Protection/Cache/Loggingを自然に入れられる
- デモで「自慢しやすい」完成物になる

### デメリット

- 画像処理が苦手な読者には心理的ハードルがある
- 実運用より「遊び」に寄って見える可能性
- ローカル画像の準備が必要

## 案B: 「秘密基地ゲートキーパー」アプローチ

### シリーズ名案

**「秘密基地ゲートキーパーを作ろう」**（全5回）

### 特徴・アプローチ

秘密基地の「ドア・金庫・監視センサー」を操作するCLIを作る。アクセス制御とログを軸に、段階的にProxyで守る。セキュリティっぽい題材だが、あくまで自己完結の仮想基地として倫理面をクリアする。

### USP（独自の価値提案）

**「なぜこのシリーズにお金を払う価値があるのか？」**

- **“アクセス制御をProxyで切り出す”という実務直結の技を最短距離で体験できる**
- **ログ/監査を絡めて「なぜProxyが必要か」を痛みとして学べる**
- **基地という設定で「ちょっと生意気」な世界観を演出できる**

### メリット

- セキュリティ文脈でProxyの必然性が強い
- 監査ログ・権限管理などSOLIDと親和性が高い
- コードがCLI中心で環境依存が少ない

### デメリット

- 既存Proxyシリーズ（攻撃シミュレータ）と近い空気感に見えるリスク
- 見た目の派手さが弱い
- 説明が堅めになりやすい

## 案C: 「宇宙ドローン遠隔操縦」アプローチ

### シリーズ名案

**「宇宙ドローン遠隔操縦盤を作ろう」**（全5回）

### 特徴・アプローチ

地球から宇宙ドローンを操縦する設定で、リモート操作・通信遅延・キャッシュ・権限制御をProxyで吸収する。通信が切れたらダミーで動かすなど、実務とゲーム性の両立を狙う。

### USP（独自の価値提案）

**「なぜこのシリーズにお金を払う価値があるのか？」**

- **リモートProxyの必然性を“通信遅延”という体験で納得できる**
- **ダミー操縦やキャッシュで「止まらないシステム」を作る視点が得られる**
- **SF設定で「完成物を見せる楽しさ」が強い**

### メリット

- リモートProxyの理解が深まる
- エラー/遅延をProxyで吸収する実践ができる
- シミュレータと本番の切り替えが学べる

### デメリット

- 仕掛けが多く、初心者には難しく感じる可能性
- テーマがSF寄りで実務感が薄いと捉えられる

## 推薦案とその理由

### 推薦：案A「ゴーストギャラリー」アプローチ

### 推薦理由

1. **ペルソナとの適合性**
   - 視覚的で分かりやすく、初学者でも「違い」が体感しやすい
   - 画像読み込み問題はWeb/CLI問わず共感しやすい

2. **検索意図との適合性**
   - 「遅延ロード」「キャッシュ」「アクセス制御」など検索されやすいキーワードが揃う
   - 作品名にパターン名を入れずに、内容でProxyを示せる

3. **学習効果**
   - 代理を段階的に増やす構成で、Proxyのバリエーションを自然に体験できる
   - 問題→Proxy導入→解決の流れが各回で明確

4. **既存シリーズとの連続性**
   - ブルートフォース攻撃系とは題材が完全に異なり、重複が少ない
   - 「ちょっと生意気」な遊び心は残しつつ、攻撃系から離れている

### 代替案の選択指針

- **案Bが適しているケース**: セキュリティ/監査に強い文脈でProxyを教えたい場合
- **案Cが適しているケース**: リモート操作やネットワーク通信が主題の読者に刺さる場合
