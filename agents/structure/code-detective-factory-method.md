# コード探偵ロックの事件簿【Factory Method】縛られた生成ライン〜newの呪縛と密室の工場〜

## 概要

- **ターゲット層**:
  - `new` キーワードで特定の具象クラスを直接インスタンス化し、コードが条件分岐（if-elsif等）だらけになっている初級〜中級のサーバーサイドエンジニア。
  - 通知処理や決済処理などで、種類（Email, SMS, Slack 等）が増えるたびに既存クラスを変更している人。
- **目的**:
  - Factory Methodパターンを用いて、インスタンス生成の責任を専用のCreator（工場）またはサブクラスに分離し、メインロジックから `new` の依存をなくす方法を理解する。
  - 「生成」と「利用」の関心の分離（Seperation of Concerns）を学ぶ。
- **公開予定日**: 2026-03-18 07:07:05 +09:00

## 設定（Phase 1: Prepare）

- **アンチパターン**: Hardcoded Instantiation（newの呪縛 / 密結合な生成）
- **解決パターン**: Factory Method パターン
- **依頼人（ワトソン役）**: マイク（若手バックエンドエンジニア）
- **舞台**: 急成長中のスタートアップのサーバー室（または依頼人の開発ブース）
- **苦境**: 「決済完了システムで通知を送る機能を作っているんですが、最初はEmailだけだったのに、SMS、Slack、Push通知と次々に追加要求が来て…。そのたびに `OrderService` クラスの奥深くにある `if ($type eq 'email') { return EmailNotifier->new() }` という条件分岐に手を入れる羽目になり、コードがスパゲッティ状態なんです。しかも他の開発者が触るとすぐ壊れます……」
- **メタファー**: 
  - 接着剤ベタベタの服（コードが `new` キーワードにまみれて密結合している様）
  - 「使う者（客）」が「おにぎりの握り方（生成方法）」まで知っている屋台（ハードコードされた生成）から、注文だけすれば専門の職人（Factory）が作ってくれるシステムへの変化。

## 構成案（統合版）

1. **導入：依頼人の訪問**
   - マイクが大量のエラートレース（またはスパゲッティ状のコードの山）を抱えてロックの事務所に現れる。
   - 服（コード）のあちこちに `new` という強力な接着剤がこびりついている描写。
   - ロックの診断：「『new』はただの演算子じゃない。最も強力な接着剤だ」
2. **事件現場（Before: アンチパターン）**
   - マイクの `OrderService.pm`（あるいはNotifier呼び出し元）のコードを提示。
   - `if ($type eq 'email') { return EmailNotifier->new() } elsif ($type eq 'sms') { return SmsNotifier->new() } ...` のような条件分岐の羅列。
   - 問題点：通知の種類が増えるたびにメイン処理を修正（Open-Closed Principle違反）。
3. **推理と解法（解決パターンの提示）**
   - ロックが「生成（誰がどのように作るか）」と「使用（どう使うか）」の責任を切り離すトリックを見せる。
   - 「Factory Method」の概念図（または比喩、例えば「ピザを注文する時に、客がオーブンの温度設定までする必要はない」など）を説明。
   - 通知（Notifier）インターフェースと、それを生成する Factory の役割分担を解説。
4. **解決編（After: リファクタリング）**
   - Perl (Moo) による Factory Method の実装例。
   - `NotifierFactory`（またはCreator役クラス）を作成し、生成ロジックを委譲。
   - メイン処理（`OrderService`）は Factory に `create('email')` と頼むだけになり、スッキリする。
   - （※実用的なPerlの実装として、モジュール名を動的に `require` する抽象ファクトリ的なアプローチ、またはハッシュテーブルでクラスをマッピングするアプローチを含める）
5. **結末：探偵の教訓**
   - マイクが新しい「LINE通知」を追加する際、メインクラスを一切触らずに Factory の設定（または新しいCreatorの追加）だけで済むことに感動する。
   - ロック特製のコーヒー（あるいは別の煙ったアイテム）で締め。
6. **探偵の調査報告書**
   - まとめ：Factory Method とは何か。いつ使うべきか。依存性逆転（DI）への繋がり（余裕があれば）。

## メトリクス比較

| 項目 | 修正前 (Before) | 修正後 (After) | 改善効果 |
|------|-----------------|----------------|----------|
| **クラス変更の波及** | `OrderService` + 個別の通知クラス | `NotifierFactory` または設定情報のみ | OCPに適合、メインロジックの安定化 |
| **if-elsif の数** | 通知種の数だけ増加 | ゼロ（動的ロード/ディスパッチ） | 循環的複雑度の大幅低下 |
| **テスト容易性** | 実クラスと密結合、モック困難 | Factoryを入れ替える/モック化することで容易 | DIへの親和性が高まる |
