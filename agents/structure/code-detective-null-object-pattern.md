## 語り部（ワトソン君）プロファイル
- **属性**: 30歳、男性、社内システム部の中堅エンジニア
- **一人称**: 僕
- **背景**: 社内の通知基盤（メール・Slack・ログ出力）を担当。ユーザー設定により通知先が異なり、「通知不要」のユーザーが全体の7割。あらゆる通知呼び出し箇所に `if (defined $notifier)` ガードが増殖し、新チャネル追加時に30箇所以上の修正が必要になっている。先週の本番障害は、1箇所だけ `defined` チェックを入れ忘れたことが原因。
- **性格**: 真面目だが要領が悪い。「念のため」のチェックを積み重ねるタイプで、防衛的プログラミングに無自覚。

## プロット（5幕構成）

### 第1幕: 導入（依頼）— 幽霊に怯える開発者
- ワトソンがLCIを訪ねる。エナジードリンクの缶が散乱するオフィス描写。
- 悩み: 「通知システムにチャネルを追加するたびに、`defined` チェックを30箇所以上に入れなきゃいけない。先週1箇所漏れて本番障害になった」
- ロックの第一声: 「ほう…幽霊に怯えているわけか」
- ワトソンのツッコミ: 「幽霊って…ただのundefですよ？」
- ロックが「undefは幽霊だよ、ワトソン君。存在しないのに、おまえたちの行動を支配している」と芝居がかって宣言。

### 第2幕: 現場検証 — Beforeコードの調査
- ロックがコードを検分。Notificationクラスの `send()` メソッド周辺。
- `if (defined $self->email_notifier)` / `if (defined $self->slack_notifier)` / `if (defined $self->logger)` が大量に並ぶ。
- ロック: 「初歩的な（コードの）においだよ、ワトソン君。同じ防衛線が何十箇所にも張られている」
- Beforeコードを提示（Perl/Moo）。
- ワトソンの心の声:（確かに、同じようなifが至る所にある…でもチェックしないとエラーになるし…）

### 第3幕: 推理披露 — Null Objectの導入
- ロック: 「犯人はundefではない。undefを恐れるおまえたちの *臆病さ* が真犯人だ」
- 解決策: 「存在しない証人を、証言台に立たせるんだ」
- NullNotifierの導入を説明。Notifierロール（インターフェース）を定義し、EmailNotifier / SlackNotifier / NullNotifier を実装。NullNotifierは全メソッドが空実装（何もしない）。
- ワトソン: 「何もしないオブジェクトを作るって…意味あるんですか？」
- ロック: 「何もしないことに *意味がある* のだよ。幽霊を消すのではない、幽霊に実体を与えるんだ」
- Afterコードを提示。`defined` チェックが全消滅。
- Mermaid図: クラス図（Notifier Role → EmailNotifier, SlackNotifier, NullNotifier）

### 第4幕: 解決 — テストの通過と秩序の回復
- 修正コードでテストがすべてグリーン。
- ワトソン: 「30箇所あった `if defined` が…全部消えた」
- ロック: 「すべての不吉な `if` 構文を排除して残ったものが、いかにオブジェクト指向的でなくとも、それが真実（デザインパターン）なんだ」
- 新チャネル追加のデモ: SmsNotifierを追加してもNotificationService側の変更はゼロ。
