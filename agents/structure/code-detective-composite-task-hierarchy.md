# コード探偵ロックの事件簿【Composite】地下組織の系譜図〜末端まで届かない指令〜

## 概要

- **ターゲット層**:
  - ツリー構造のデータ（タスク階層、カテゴリ、組織図など）を扱う際に、ノード種別ごとの `ref` / `instanceof` 分岐が全操作に散在して保守に苦しんでいる初級〜中級エンジニア。
  - 新しいノード種を追加するたびに既存の関数をすべて修正しなければならない状況に直面している人。
- **目的**:
  - Compositeパターンを用いて「個」と「集合」を同一インターフェースで扱い、ツリー構造を再帰的かつ統一的に操作する方法を理解する。
  - 型チェック分岐を排除し、開放閉鎖原則（OCP）を自然に実現する設計を体験する。
- **公開予定日**: 2026-03-24 07:07:05 +09:00

## 設定（Phase 1: Prepare）

- **アンチパターン**: Type Check Dispatch（型チェック分岐の散在 / ref によるノード種別判定の繰り返し）
- **解決パターン**: Composite パターン
- **依頼人（ワトソン役）**: 高橋（中堅SIerのエンジニア・男性・32歳・経験8年）。自社PMツール「TaskFlow」の保守を前任者から引き継いだ。実力はあるが、ツリー構造の再帰処理に手を焼いている。一人称は「私」。
- **舞台**: 中堅SIerの自社プロダクト。元はフラットなタスクリストだったPMツールに、サブタスク・孫タスクが追加され、進捗集計コードが型チェックだらけに。
- **苦境**: 「進捗率を出すだけの関数なのに、階層が深くなるとバグるんです。しかも今度マイルストーン機能を追加しろと言われて……全関数にif文を追加するんですか？」
- **メタファー**:
  - ツリー構造 = 地下組織の系譜図。ボス（ルート）、幹部（コンポジット）、末端構成員（リーフ）。
  - 型チェック分岐 = 役職ごとにバラバラの指令手順。新しい役職が増えると全手順書を改訂。
  - Composite = 統一された命令系統。ボスも構成員も同じ形式で「報告せよ」に応答する。

## 構成案（統合版）

1. **導入：依頼人の訪問**
   - 高橋がタスク階層のツリー図を印刷した紙を持ってLCIに駆け込む。
   - ロック：「これは地下組織の系譜図と同じ構造だよ、ワトソン君」
   - 高橋が「サブタスクのネストが深くなると進捗計算がバグる。マイルストーン追加で全関数に手を入れる必要がある」恐怖を語る。

2. **現場検証（Before: アンチパターン）**
   - `Task`（リーフ）と `TaskGroup`（グループ）の2種類のノード。
   - `total_hours()`、`count_completed()`、`count_all()` の各関数で `if (ref $node eq 'Task') { ... } elsif (ref $node eq 'TaskGroup') { ... }` が毎回繰り返される。
   - ロック：「ボスへの報告と末端への命令がまったく別の手順になっている。こんな組織は統率がとれていない」
   - Milestoneを渡すと `Unknown node type` で死ぬデモ。

3. **推理披露（After: Compositeパターンの導入）**
   - `TaskComponent` ロール（共通インターフェース）を定義。
   - `TaskLeaf`（末端構成員）と `TaskComposite`（幹部）が同じ掟に従う。
   - ロック：「組織のどの階層にも同じ命令系統が通る」
   - `Milestone` の追加デモ：Roleを実装するだけ、既存コード修正ゼロ。
   - Mermaid図でクラス構造を可視化。

4. **解決：テストの通過**
   - テストが全て通過。3階層ネスト＋Milestoneでも正しく動作。
   - ロック：「組織の深さに上限はない。これがCompositeの再帰的な強さだ」
   - 報酬ジョーク：「組織の階層の深さと同じ杯数のエスプレッソ」。

5. **探偵の調査報告書**
   - 容疑：Type Check Dispatch（型チェック分岐の散在）
   - 真実：Composite パターン
   - 証拠：型チェック if 文の完全消滅、新ノード追加はクラス1つ、既存コード修正ゼロ
   - 推理のステップ（4項目）
   - ロックより（メッセージ）：Compositeの本質と適用の注意点

## メトリクス比較

| 項目 | 修正前 (Before) | 修正後 (After) | 改善効果 |
|------|-----------------|----------------|----------|
| **ノード種別の追加** | 全操作関数に `elsif` 分岐を追加（O(N)箇所の修正） | Roleを実装するクラスを1つ追加するだけ | 既存コード修正ゼロ（OCP準拠） |
| **型チェック if 文** | 操作の数 × ノード種の数だけ散在 | 完全に消滅 | ランタイムエラーのリスク解消 |
| **ツリーの深さ** | 型チェックが正しければ動作するが、忘れるとクラッシュ | 深さに制限なく統一的に動作 | 再帰的な安全性 |
