# 連載構造案：Template Methodパターンを学ぶ新シリーズ

調査結果: `content/warehouse/template-method-pattern.md` に基づく

## 前提情報

- **技術スタック**: Perl v5.36以降（signatures対応）、Mooによるオブジェクト指向プログラミング
- **想定読者**: Perl入学式卒業したばかりの入門者
- **想定ペルソナ**: 「Mooで覚えるオブジェクト指向プログラミング」シリーズを読了してオブジェクト指向プログラミングを身に付けたい、実務で役立つモダンなPerlを使ってみたい
- **学習目標**:
  - オブジェクト指向プログラミングの原則を深く学べる
  - SOLID原則（特に開放閉鎖原則OCP）を深く学べる
  - 自然にTemplate Methodパターンを覚えられる
  - デザインパターンの名前が覚えられる
  - Hollywood Principle（制御の反転）を体験できる
- **位置づけ**: 「Mooで覚えるオブジェクト指向プログラミング」シリーズの続編、「ディスパッチャー」「データエクスポーター」と並ぶデザインパターンシリーズ
- **ストーリー**: 知っている知識で動くプログラムを作成 → 同じ処理手順が増えてコピペ地獄 → 継承で処理の骨格を共通化して問題を解決
- **難易度**: 3/5（継承、抽象メソッド、Hollywood Principleの理解が必要）
- **制約**:
  - 1記事1概念
  - コード例2つまで
  - 回の最後には完成コードを示す（原則1つのスクリプトファイル）
  - **デザインパターンの名前はシリーズ名に敢えて出さない**（最終回で明かす）

### 既存シリーズとの差別化

以下のシリーズとは完全に異なる題材を使用：

| シリーズ | 題材 | パターン | アプローチ |
|---------|------|---------|-----------|
| ディスパッチャーを作ろう | URLルーティング | Strategy | 委譲（has + Role） |
| データエクスポーターを作ろう | CSV/JSON/YAML出力 | Strategy | 委譲（has + Role） |
| 通知システムを作ろう | 通知方法切り替え | Strategy | 委譲（has + Role） |
| レポートジェネレーター | レポート生成 | Factory Method | 継承（extends + オーバーライド） |
| DB接続管理 | DB接続 | Factory Method | 継承（extends + オーバーライド） |
| ゲームキャラクター生成 | キャラ生成 | Factory Method | 継承（extends + オーバーライド） |
| **本シリーズ（新規）** | **別のドメイン** | **Template Method** | **継承（処理の骨格定義）** |

**Template Methodパターンの特徴**:

- **処理の骨格を定義**: 基底クラスで「手順」を固定
- **継承ベース**: サブクラスがオーバーライドして一部のステップを実装
- **Primitive Operations**: 必ず実装するステップ（`die "must implement"`）
- **Hook Methods**: オプションで上書き可能なステップ（デフォルト実装）
- **Hollywood Principle**: "Don't call us, we'll call you"（制御の反転）
- **典型的なユースケース**: データ処理パイプライン、ドキュメント生成、ビルドプロセス、ゲームのターン処理

### 前提知識（前シリーズで習得済み）

| 前シリーズで学んだこと | 本シリーズでの活用 |
|----------------------|-------------------|
| `has`と`sub`でクラスを定義 | 基底クラスとサブクラスの定義 |
| `new`でオブジェクト生成 | 各プロセッサー/コンバーターのインスタンス化 |
| `is => 'ro'/'rw'`でアクセス制御 | 属性の定義 |
| `required`と`default` | 入力データの初期化 |
| **`extends`による継承** | **★最重要：基底クラスの継承** |
| **オーバーライド** | **★最重要：Primitive Operations/Hook Methodsの実装** |

## 案A: 「Webスクレイピングパイプライン」アプローチ

### シリーズ名案

**「Perlを使ってWebスクレイパーを作ってみよう」**（全10回）

### 特徴・アプローチ

ニュースサイト、EC サイト、天気予報サイトなど異なるWebサイトから情報を収集する「Webスクレイパー」を作成します。すべてのスクレイパーで共通の処理手順（取得→抽出→整形→保存）を基底クラスで定義し、サイトごとの具体的な抽出ロジックをサブクラスで実装します。

**Template Methodパターンとの対応**:

| パターン要素 | 本シリーズでの実装 |
|------------|------------------|
| Abstract Class（抽象クラス） | WebScraper基底クラス |
| Template Method（テンプレートメソッド） | scrape()メソッド（取得→抽出→整形→保存の順で呼び出す） |
| Primitive Operations（基本操作） | extract_data()（サイトごとに異なる抽出ロジック） |
| Hook Methods（フックメソッド） | validate_data(), log_result()（デフォルト実装あり） |
| Concrete Classes（具象クラス） | NewsScraper, WeatherScraper, ProductScraper等 |

### USP（独自の価値提案）

**「なぜこのシリーズにお金を払う価値があるのか？」**

- **実用性が極めて高い**: Webスクレイピングは実務で頻出（価格監視、競合調査、データ収集）。無料チュートリアルは「とりあえず動く」レベルだが、本シリーズは**拡張性・保守性を重視した設計**を学べる
- **「ハッキング的」な魅力**: 友人に「Perlで自動的にWebサイトから情報を抜き出すツールを作った」と自慢できる少し生意気なテーマ
- **デザインパターンの必然性**: 同じ手順（取得→抽出→保存）を複数サイトで繰り返すため、Template Methodパターンの価値が**体感**できる
- **モダンPerl**: Mojo::UserAgent（MojoliciousのHTTPクライアント）を使い、v5.36のシグネチャでスッキリ書ける
- **段階的な困難の導入**: 最初は1サイトだけスクレイピング→2サイト目でコピペ地獄→継承で解決という自然な流れ

### メリット

- **実用的**: 価格監視、ニュース収集、天気情報取得など実務で使えるスキル
- **達成感**: 「Webから自動的にデータを取得できた！」という明確な成果
- **Template Methodの必然性が高い**: 「取得→抽出→保存」という共通手順が複数サイトで繰り返されるため、パターンの価値が自然に理解できる
- **モダンPerlの魅力**: Mojo::UserAgentでクールなコード
- **拡張性のデモが容易**: 新しいサイトを追加する際に既存コードを触らないOCPの実証

### デメリット

- **CPANモジュール依存**: Mojolicious (Mojo::UserAgent)、Mojo::DOMが必要
- **外部サイト依存**: 実際のWebサイトの構造変更でコードが動かなくなるリスク（→サンプルHTMLで対応可能）
- **倫理的配慮が必要**: Webスクレイピングのマナー（robots.txt、rate limiting）の説明が必要

### 差別化ポイント

- **Webスクレイピング**: 既存シリーズにない実用的テーマ
- **「ハッキング的」要素**: 友人に自慢できる少し生意気な題材
- **継承による処理の骨格定義**: Strategyパターン（委譲）やFactory Method（生成）とは異なる視点
- **Hollywood Principle体験**: 基底クラスがサブクラスを「呼び出す」制御の反転を実感

## 案B: 「ファイル変換パイプライン」アプローチ

### シリーズ名案

**「Perlでファイル変換ツールを作ってみよう」**（全10回）

### 特徴・アプローチ

Markdown→HTML、画像リサイズ、ログファイル整形など、異なる形式のファイル変換を行う「ファイルコンバーター」を作成します。すべてのコンバーターで共通の処理手順（読み込み→変換→書き込み）を基底クラスで定義し、形式ごとの具体的な変換ロジックをサブクラスで実装します。

**Template Methodパターンとの対応**:

| パターン要素 | 本シリーズでの実装 |
|------------|------------------|
| Abstract Class（抽象クラス） | FileConverter基底クラス |
| Template Method（テンプレートメソッド） | convert()メソッド（読み込み→変換→書き込みの順で呼び出す） |
| Primitive Operations（基本操作） | transform_content()（形式ごとに異なる変換ロジック） |
| Hook Methods（フックメソッド） | validate_input(), cleanup()（デフォルト実装あり） |
| Concrete Classes（具象クラス） | MarkdownConverter, ImageResizer, LogFormatter等 |

### USP（独自の価値提案）

**「なぜこのシリーズにお金を払う価値があるのか？」**

- **実務で即戦力**: ドキュメント生成、画像一括処理、ログ解析など実務で頻出する処理
- **汎用性が高い**: 「読み込み→変換→書き込み」というパターンはあらゆる変換処理に応用可能
- **Template Methodの必然性**: 変換処理の骨格は共通だが、変換ロジックだけが異なるという典型的なユースケース
- **モダンなファイル処理**: Path::Tiny（モダンなファイルI/O）、Text::Markdown（Markdown変換）など便利なモジュールを活用
- **拡張性**: 新しい変換処理を追加する際に既存コードを触らないOCPの実証

### メリット

- **実用性が極めて高い**: ドキュメント生成、画像処理、ログ整形は実務で頻出
- **CPANモジュールの活用**: Path::Tiny、Text::Markdownなど便利なモジュールの使い方を学べる
- **Template Methodの自然な導入**: 「読み込み→変換→書き込み」という共通手順が明確
- **外部依存が少ない**: 実際のWebサイトに依存しないため、動作が安定

### デメリット

- **やや地味**: Webスクレイピングと比べて「ハッキング的」な魅力に欠ける
- **CPANモジュール依存**: Text::Markdown、Image::Magick（重い）などが必要
- **画像処理の複雑さ**: Image::Magickのセットアップが環境によって難しい

## 案C: 「自動デプロイスクリプト」アプローチ

### シリーズ名案

**「Perlで自動デプロイツールを作ってみよう」**（全10回）

### 特徴・アプローチ

開発環境、ステージング環境、本番環境など異なる環境へのデプロイを行う「デプロイスクリプト」を作成します。すべてのデプロイで共通の処理手順（準備→ビルド→テスト→デプロイ→後処理）を基底クラスで定義し、環境ごとの具体的な設定やデプロイ方法をサブクラスで実装します。

**Template Methodパターンとの対応**:

| パターン要素 | 本シリーズでの実装 |
|------------|------------------|
| Abstract Class（抽象クラス） | Deployer基底クラス |
| Template Method（テンプレートメソッド） | deploy()メソッド（準備→ビルド→テスト→デプロイ→後処理の順で呼び出す） |
| Primitive Operations（基本操作） | deploy_to_server()（環境ごとに異なるデプロイ方法） |
| Hook Methods（フックメソッド） | before_deploy(), after_deploy()（デフォルト実装あり） |
| Concrete Classes（具象クラス） | DevDeployer, StagingDeployer, ProductionDeployer等 |

### USP（独自の価値提案）

**「なぜこのシリーズにお金を払う価値があるのか？」**

- **エンジニアリング的な魅力**: 「自動デプロイツールを自作した」という経験は友人に自慢できる
- **実務で即戦力**: CI/CD、自動化スクリプトは現代の開発現場で必須スキル
- **Template Methodの必然性**: デプロイ手順は共通だが、環境ごとに設定が異なるという典型的なユースケース
- **ハッキング的な楽しさ**: コマンドライン、Git操作、SSH接続など「エンジニアっぽい」処理を扱う
- **拡張性**: 新しい環境を追加する際に既存コードを触らないOCPの実証

### メリット

- **エンジニアリング的な魅力**: デプロイ自動化は「プロっぽい」スキル
- **実務で役立つ**: CI/CD、自動化は現代の開発現場で必須
- **Template Methodの自然な導入**: デプロイ手順（準備→ビルド→テスト→デプロイ→後処理）が明確
- **「ハッキング的」要素**: Git、SSH、コマンドライン操作など生意気な雰囲気

### デメリット

- **環境構築の複雑さ**: Git、SSH設定が必要で初心者にはハードル高い
- **実際のサーバーが必要**: 実環境でのテストが難しい（→ローカルでのシミュレーション可能）
- **セキュリティ配慮**: SSH鍵、パスワード管理の説明が必要
- **やや上級者向け**: デプロイの概念自体を理解する必要がある

## 推薦案とその理由

### 推薦：案A「Webスクレイピングパイプライン」アプローチ

### 推薦理由

1. **ペルソナとの適合性**
   - 「Mooで覚えるオブジェクト指向プログラミング」で学んだ継承とオーバーライドを実践的に活用できる
   - Webスクレイピングという具体的な成果物があり、達成感を得やすい
   - 実務でも使える知識として学習意欲が高まる

2. **検索意図との適合性**
   - 「Perl Webスクレイピング」「Perl Mojo」などの検索でたどり着く可能性がある
   - Template Methodパターンの典型例（データ処理パイプライン）として紹介されることが多いドメイン
   - 実用性が高く、他の記事との差別化が明確

3. **学習効果**
   - 継承とオーバーライドの実践的な使い方を体験できる
   - 「コピペが増えて困る → 継承で解決」という問題解決のストーリーが明確
   - Strategyパターン（委譲ベース）との違いが理解しやすい
   - Hollywood Principleを体験できる

4. **既存シリーズとの差別化**
   - **Webスクレイピング**: 既存シリーズ（URLルーティング、CSV/JSON/YAML出力、通知システム、レポート生成、DB接続、ゲームキャラクター）のどれとも異なるドメイン
   - **処理の骨格定義**: Strategyパターン（委譲）やFactory Method（生成）とは異なり、Template Methodは「処理の手順」に焦点
   - **ハッキング的要素**: 友人に自慢できる少し生意気な題材

5. **内部リンクの活用**
   - 「Mooで覚えるオブジェクト指向プログラミング」第8回（継承）、第9回（オーバーライド）と密接にリンク可能
   - 「ディスパッチャーを作ろう」第12回でのStrategyパターン解説と対比できる

6. **USPの強さ**
   - 無料チュートリアルは「とりあえず動く」レベルだが、本シリーズは**拡張性・保守性を重視した設計**を学べる
   - 「Perlで自動的にWebサイトから情報を抜き出すツールを作った」と友人に自慢できる

### 代替案の選択指針

- **案B（ファイル変換パイプライン）が適しているケース**:
  - より安定した環境で学習したい場合（外部サイトへの依存を避けたい）
  - ドキュメント生成、画像処理など汎用的な変換処理に興味がある読者層を想定する場合
  - Path::Tiny、Text::Markdownなど便利なモジュールの使い方も学びたい場合

- **案C（自動デプロイスクリプト）が適しているケース**:
  - より上級者向けのエンジニアリング的なテーマで魅力を打ち出したい場合
  - CI/CD、自動化スクリプトに興味がある読者を対象とする場合
  - 「プロっぽい」スキルを強調したい場合
