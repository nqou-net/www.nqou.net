# 連載構造案：本棚アプリで覚える集合体の巡回

調査結果: `content/warehouse/iterator-pattern-research.md` に基づく

## 前提情報

- **技術スタック**: Perl v5.36以降（signatures対応）/ Moo
- **想定読者**: Perl入学式卒業したばかりの入門者
- **想定ペルソナ**: 「Mooで覚えるオブジェクト指向プログラミング」シリーズを読了してオブジェクト指向プログラミングを身に付けたい読者
- **学習目標**: Iteratorパターンの4要素（Iterator、ConcreteIterator、Aggregate、ConcreteAggregate）を段階的に理解し、自然にデザインパターンの概念を習得する
- **位置づけ**: 「Mooで覚えるオブジェクト指向プログラミング」シリーズの続編
- **ストーリー**: 知っている知識でプログラムを作成→機能追加で問題発覚→デザインパターンで解決
- **シリーズ名の制約**: デザインパターン名（Iterator、イテレータ）はシリーズ名に**含めない**

### 前提知識（前シリーズで習得済み）

| 前シリーズで学んだこと | 本シリーズでの活用 |
|----------------------|-------------------|
| `has`による属性定義 | 本棚やイテレータの状態管理 |
| `is => 'ro'` / `is => 'rw'` | 読み取り専用属性の設計 |
| `required` / `default` | 初期位置の設定 |
| カプセル化（内部属性の隠蔽） | 集合体の内部構造を隠す |
| 複数クラスの連携 | Iterator と Aggregate の関係 |
| ロール（`Moo::Role`） | Iterator インターフェースの定義 |
| 委譲（`handles`） | 走査機能の委譲 |

### Iteratorパターンの4要素（学習順序）

1. **Aggregate（集合体）**: コレクションを持つクラス（例: BookShelf）
2. **Iterator（反復子）**: 走査のインターフェース（ロールで定義）
3. **ConcreteIterator（具体的な反復子）**: 実際の走査ロジック
4. **ConcreteAggregate（具体的な集合体）**: イテレータを返すメソッドを持つ

## 案A: 「本棚アプリで覚える集合体の巡回」アプローチ

### 特徴・アプローチ

GoFの「Design Patterns」でも使われている定番の「BookShelf（本棚）」を題材に、シンプルで分かりやすいIteratorパターンの学習を行います。問題解決駆動型で、まずfor文による直接的なアクセスの問題点を体験してから、段階的にパターンを導入します。

### メリット

- GoFのオリジナル例題と同じ題材で、後から参考資料を調べやすい
- 「本を1冊ずつ取り出す」という直感的なメタファー
- シンプルな構造で理解しやすい
- 前シリーズ（掲示板）とは異なる題材で新鮮味がある

### デメリット

- やや古典的な例題で新鮮味に欠ける可能性
- 実用性を感じにくい読者もいるかもしれない

- 音楽アプリのイメージで楽しく学べる

### デメリット

- 本棚に比べてやや複雑になる

### メリット

- 実際のシステム開発を想起させる実践的な題材
- 複数の集合体（書棚、書庫、貸出中）で統一インターフェースの価値を実感
- より本格的なアプリケーション設計の雰囲気

### デメリット

- 案A/Bに比べて複雑で、入門者には敷居が高い
- 複数ファイル構成が必要になる可能性
- 5回では収まりきらない可能性

## 推薦案とその理由

### 推薦：案A「本棚アプリで覚える集合体の巡回」

### 推薦理由

1. **ペルソナとの適合性**
   - Perl入学式卒業直後の入門者には、シンプルで理解しやすい題材が最適
   - 案Bのシャッフル機能や案Cの複数集合体は、Iteratorパターン以外の複雑さを持ち込むリスク
   - 案Aは「本を1冊ずつ取り出す」という直感的なメタファーで理解を助ける

2. **検索意図との適合性**
   - 「Perl Iterator パターン」で検索した読者が、GoFの原著と同じ例題で学べる
   - 学習後に英語の参考資料（GoF原著、Refactoring Guru等）を読む際のハードルが下がる
   - 「本棚」という普遍的な題材は時代を問わず理解できる

3. **学習効果**
   - 問題解決駆動型で「なぜIteratorパターンが必要か」が明確
   - 1記事1概念の原則を守りやすいシンプルな構成
   - 第5回で「実は作っていたものにはIteratorパターンという名前がある」という発見の喜び

4. **既存シリーズとの連続性**
   - 「Mooで覚えるオブジェクト指向プログラミング」で学んだロール（Moo::Role）を自然に活用
   - 前シリーズ（掲示板）とは異なる題材で新鮮味を保ちつつ、技術スタックは一貫
   - 「問題解決駆動」アプローチを引き継ぎ、学習スタイルの統一

5. **内部リンクの活用**
   - 第3回でロールを使う際、前シリーズの該当記事への内部リンクを設置可能
   - 調査ドキュメントの「本棚」実装例をほぼそのまま活用可能

### 代替案の選択指針

- **案Bが適しているケース**: 読者が若年層中心で、より身近な題材を求める場合。複数イテレータの価値を強調したい場合。
- **案Cが適しているケース**: 読者がすでにある程度の開発経験があり、実践的なシステム設計を学びたい場合。ポリモーフィズムの価値を特に強調したい場合。
