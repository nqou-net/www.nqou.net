## 2. 物語設定（Phase 0 合意事項）
- **容疑者（アンチパターン）**: God Class（万能すぎるコントローラー、密結合なサブシステム操作の乱立）
- **真実（解決パターン）**: Facadeパターン（複雑な内部を隠すシンプルな窓口）
- **ワトソン君（依頼人）**: 休日にカフェでくつろいでいたところを前職の先輩に強引に拉致された「巻き込まれ被害者」。
- **テーマ**: TMTOWTDI（やり方はひとつじゃない）。同じ神オブジェクトという病理でも、症状によって処方箋（適用するパターン）は異なる。

## 3. プロット構成（統合版）

### 導入（[起] 誘拐と地雷原）
- 休日、ワトソン君がカフェで技術書を読んでいるシーンからスタート。
- 突然、オラオラ系の豪腕PMである「前職の先輩」が乱入。
- 「お前いま暇だろ！システムが動かなくなった、手伝え！あとあの凄腕の探偵（ロック）も呼べ！」と無茶振りされ、別の現場のプロジェクトルームへ強制連行される。

### 現場検証（[承] 絡みつくサブシステムの森）
- 現場のコードを見ると、決済、在庫引当、通知、ログ出力といったあらゆるサブシステムを `GodManager` または `OrderScript` などが**すべて直接**初期化・操作している。
- 「俺が全部把握して直接指示を出せば最速だ」という先輩の自負。
- しかし外部APIの仕様変更が入り、初期化順序が少し狂っただけで全体がクラッシュする「パンドラの箱」になっていた。
- 先輩が力技で直そうとするたびに、別のサブシステムからエラー火柱が上がる。

### 推理披露（[転] 窓口の設立とTMTOWTDI）
- ロックが到着。以前の事件（第1回：Strategy）との対比に言及。
- ワトソン君「前と同じ神クラスですね。Strategyパターンで切り分けますか？」
- ロック「同じ神の病でも、症状が違えば処方箋も変わる。TMTOWTDI（There's More Than One Way To Do It）。それが設計の美しいところだ」
- ロックの解説：「今回は『振る舞いの分岐』ではなく、『複雑すぎる配線』が問題だ。利用者にすべての配線を操作させるのではなく、裏に隠して『ボタン一つの窓口』だけを見せればいい」
- ロックが `OrderFacade` クラスを作成し、複雑なサブシステム群をその奥に隠蔽する実演（Before/Afterコード）。

### 解決（[結] 事件の終わり）
- メインのスクリプトは、Facadeのメソッドを1〜2個呼ぶだけの劇的なシンプルさに変貌。
- 順序依存や複雑なエラー処理はすべてFacade内で完結し、テストも無事に通る。
- 先輩はその鮮やかさに唖然。（もしかして俺のやり方が間違っていたのか…？）
- 帰り道、ロックとワトソン君の会話。Facadeはただのラップだが、システムの平和を守る重要な防壁であることを語る。
