# コード探偵ロックの事件簿【Builder】果てしなき引数の行列〜コンストラクタの窒息事件〜

## 概要

- **ターゲット層**:
  - コンストラクタやファクトリ関数の引数が10〜20個以上に膨れ上がり、呼び出し側のコードが可読性ゼロになっている初級〜中級エンジニア。
  - 設定オブジェクトやリクエストオブジェクトの生成時に、引数の順序ミスや指定漏れでバグが頻発している人。
- **目的**:
  - Builderパターンを用いて、複雑なオブジェクトの段階的な組み立てプロセスを提供し、可読性と安全性を飛躍的に向上させる方法を理解する。
  - 「何を設定しているか」が一目でわかるメソッドチェーンの美しさを体験する。
- **公開予定日**: 2026-03-21 07:07:05 +09:00

## 設定（Phase 1: Prepare）

- **アンチパターン**: Telescoping Constructor（引数地獄 / 伸び続けるコンストラクタ）
- **解決パターン**: Builder パターン
- **依頼人（ワトソン役）**: ケン（若手Webエンジニア・男性・20代後半）。広告配信プラットフォームの開発チームに配属されたばかり。意気込みは強いが、レガシーコードに圧倒されている。一人称は「僕」。
- **舞台**: 広告テック企業のオフィス。キャンペーン設定の生成コードが引数の洪水で溺れている。
- **苦境**: 「広告キャンペーンの設定オブジェクトを生成するコードを任されたんですが、`new` の引数が15個もあって、順番を1つ間違えるだけで全く別のキャンペーンが走ってしまうんです。先週、テスト用のキャンペーンが本番で10万円分の広告を配信してしまって……上司に殺されかけました」
- **メタファー**:
  - 引数の行列 = 何時間も並ぶ行列。順番を間違えたら最初からやり直し。
  - Builder = 窓口係。必要なものを一つずつ聞き取って、最後にまとめて発行してくれる受付カウンター。

## 構成案（統合版）

1. **導入：依頼人の訪問**
   - ケンが引数の一覧を印刷した紙（極めて長い巻物のような）を持ってLCIに駆け込む。
   - ロック：「その巻物は何だね？ まさか君のコードの引数リストか？ ……古代エジプトのパピルスかと思ったよ」
   - ケンが「引数の順番を1つずつ間違えるたびに本番事故が起きる」恐怖を語る。

2. **現場検証（Before: アンチパターン）**
   - `Campaign->new(title => ..., budget => ..., start_date => ..., end_date => ..., target_age_min => ..., target_age_max => ..., ...)` という15行に渡る巨大な `new` 呼び出し。
   - 問題点：引数の順序依存、オプショナル引数の管理不能、可読性ゼロ、バリデーション不在。
   - ロック：「このコンストラクタは窒息している。15もの引数を丸呑みにさせられて、消化不良を起こしているのさ」

3. **推理披露（After: Builderパターンの導入）**
   - ロックが `CampaignBuilder` クラスを作成。
   - 各設定項目を1つずつメソッドとして提供し、メソッドチェーンで流暢に組み立てる。
   - `CampaignBuilder->new->title("春の新生活")->budget(50000)->start_date("2026-04-01")->build()` の美しさ。
   - `build()` メソッド内でバリデーションを集約し、不正な組み合わせを事前に弾く。
   - ケン：「引数を覚えなくていいし、何を設定してるか読めば分かる！」

4. **解決：テストの通過**
   - テストが全て通過し、バリデーションエラーのテストケースも含めて安全性が証明される。
   - ロック：「もう引数の行列に並ぶ必要はない。窓口（Builder）が一つずつ丁寧に受け付けてくれるのだからね」
   - ケンが感動するも、最後に「じゃあこのBuilderも他のキャンペーン用にコピペして……」と言いかけてロックに叱られるオチ。

5. **探偵の調査報告書**
   - 容疑：Telescoping Constructor（引数地獄）
   - 真実：Builder パターン
   - 証拠：可読性の向上、バリデーションの集約、段階的構築の安全性
   - 推理のステップ（4〜5項目）
   - ロックより（メッセージ）

## メトリクス比較

| 項目 | 修正前 (Before) | 修正後 (After) | 改善効果 |
|------|-----------------|----------------|----------|
| **引数の可読性** | 15個の位置引数、順序依存 | 名前付きメソッドチェーン | 何を設定しているか一目瞭然 |
| **バリデーション** | なし（不正値でもオブジェクト生成可能） | `build()` で集約チェック | 不正な状態のオブジェクト生成を防止 |
| **拡張性** | 新しい属性追加時に全呼び出し箇所を修正 | メソッドを1つ追加するだけ | 既存コードへの影響なし |
