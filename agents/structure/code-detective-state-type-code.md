# 構造案: コード探偵ロックの事件簿【State】顔のない変装犯〜ステータスコードの呪い〜

## 1. 基本情報

- **シリーズ**: コード探偵ロックの事件簿
- **スラッグ**: `code-detective-state-type-code`
- **対象読者**: オブジェクト指向の基本は知っているが、状態分岐（ステータスコード）の管理に苦労している中級者
- **技術スタック**: Perl (v5.34+), Moo

## 2. 物語設定（Phase 0）

- **アンチパターン（容疑者）**: Type Code（ステータスマジックナンバーへの執着 / 巨大な if・switch 妖怪）
- **推理の切り札**: **State パターン**
- **ワトソン君のプロファイル**:
  - **名前**: カイト（入社3年目のバックエンドエンジニア）
  - **苦境**: ECサイトのオーダーステータス（0: 未決済, 1: 準備中, 2: 発送済, 3: キャンセルなど）が「単なる整数値」で管理されており、システム中のあらゆる場所に `if ($status == 2)` のような分岐が散乱。最近の仕様追加（「発送済からキャンセルは不可だが、準備中なら可能」など）で条件が複雑化し、「発送済みなのにキャンセル処理が通ってしまった」というバグの修正に追いまくられている。
- **事件メタファー**: **「顔のない変装犯（The Faceless Disguise）」**
  ステータスコードという「単なる数字」が、システムの都合に合わせて都合よく顔（振る舞い）を変え、あらゆる場所に現れてはバグを仕掛ける厄介な存在。
- **物語のフック**:
  今回は「ワトソン君が自ら書いたコード」ではなく、「前任者が残した負の遺産（マジックナンバーの塊）」に立ち向かう構図。探偵ロックが、「状態」という概念を単なる `if` の条件（数字）から、「自分自身で判断と振る舞いができるオブジェクト（本物の役者）」へと昇華させるパラダイムシフトを描く。

## 3. 連載構造表（今回は1記事の統合版）

| 構成 | 内容 | 見出し案 |
| --- | --- | --- |
| 導入 | カイトが前任者の残した「顔のない変装犯」に悩まされ、事務所を訪れる | （プロローグ） |
| 現場検証 | 巨大な `if - elsif` で状態ごとに振る舞いを変えている `Order` クラスの確認 | 現場検証：顔のない変装犯 |
| 推理披露 | Stateパターンを使って、各状態を独立したクラスに切り出す。マジックナンバーの排除と、状態遷移の責任を持たせる。 | 推理披露：本物の役者への昇華 |
| 解決 | テストが通り、状態ごとの分岐が消滅して安全に状態遷移できることを確認する。 | 事件の終わり：消えた数字たち |
| まとめ | アンチパターンとStateパターンの対応表、ロックからのメッセージ | 探偵の調査報告書 |

## 4. コード例の設計

### コード例1（問題版：Type Code）

- ファイル: `agents/tests/code-detective-state-type-code/lib/example1_problem.pl`
- 概要:
  - `Order` クラスが `$status` という整数プロパティを持つ。
  - `cancel()` メソッドの中で、`if ($status == 0) { ... } elsif ($status == 1) { ... } elsif ($status == 2) { die "Cannot cancel" }` のような巨大な判定を行っている。
  - `ship()` メソッドも同様の分岐を持つ。

### コード例2（改善版：State パターン）

- ファイル: `agents/tests/code-detective-state-type-code/lib/example1_solution.pl`
- 概要:
  - `OrderState::Role` インターフェースを作成（`cancel()`, `ship()` などのメソッドを要求）。
  - `OrderState::Unpaid`, `OrderState::Preparing`, `OrderState::Shipped`, `OrderState::Cancelled` などの状態クラスを作成。各クラスが自分自身の状態遷移ルールを持つ。
  - `Order` クラスは `$state` オブジェクトを持ち、委譲によって振る舞いを決定する。巨大な `if` 文は完全に消滅する。
