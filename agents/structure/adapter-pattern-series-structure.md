# 連載構造案：複数サービス連携ツールで覚えるPerl

調査結果: `content/warehouse/adapter-pattern.md` に基づく

## 前提情報

- **技術スタック**: Perl v5.36以降 / Moo / signatures
- **想定読者**: Perl入学式卒業程度、「Mooで覚えるオブジェクト指向」読了者
- **想定ペルソナ**: モダンなPerlで実用的なプログラムを書きたい入門者
- **学習目標**: Adapterパターンを自然に理解し、名前を覚える
- **位置づけ**: 「Mooで覚えるオブジェクト指向」の発展コース
- **ストーリー**: 既存のツールを活用したい → インターフェースが合わない → Adapterで橋渡し
- **制約**: 1記事1概念、コード例2つまで

### 前提知識（前シリーズで習得済み）

| 前シリーズで学んだこと | 本シリーズでの活用 |
|----------------------|-------------------|
| Mooによるクラス定義 | 既存クラスとAdapterクラスの作成 |
| has によるアトリビュート | オブジェクトの委譲先（adaptee）の保持 |
| メソッドの定義 | 新旧インターフェースのメソッド実装 |
| コンストラクタと依存注入 | Adapterへの既存オブジェクト注入 |
| handles による委譲 | Adapterパターンの基盤となる委譲の理解 |
| ロール (Role) | 共通インターフェースの定義（Target） |

### Adapterパターンの学習ポイント

- **本質**: 互換性のないインターフェースを持つクラスを接続できるようにする
- **キーワード**: 委譲（has-a関係）、インターフェース変換、既存コードの再利用
- **学習の流れ**: 既存ツールを使いたい → インターフェース不一致 → Adapterで解決
- **最終回**: パターン名を明かし、GoFデザインパターンとして位置づける
- **別名**: Wrapperパターン（既存オブジェクトを包み込む）

## 案A: 「天気情報ツールを作ってみよう」アプローチ

### 特徴・アプローチ

複数の「天気情報サービス」を統一インターフェースで扱うツールを作成する。各サービスが異なるAPI形式（メソッド名、戻り値形式）を持っており、それをAdapterで統一する。現実世界の「複数API統合」シナリオに近く、Adapterパターンの典型的なユースケースを体験できる。

### メリット

- 「天気情報」は身近で初心者にもイメージしやすいテーマ
- 複数のサービスを統一するシナリオはAdapterの典型例
- 「同じことをしたいのにAPIが違う」という問題が直感的に理解できる
- 将来的に実際のWeb APIとの連携にも応用できる

### デメリット

- 天気サービスをモックで作成するため「本物感」がやや薄い
- 複数クラスが登場するため、初心者には少し複雑に感じる可能性
- 天気情報の取得自体に興味がない読者にはモチベーションが低い

- **複数API統合**: 実務で頻出する「複数サービスの統一」シナリオを体験
- **段階的な拡張**: 1つ目のサービス → 2つ目で問題発生 → Adapter → 3つ目でも適用
- **身近なテーマ**: 天気情報は日常的で親しみやすい

## 案B: 「ログ出力ツールを作ってみよう」アプローチ

### 特徴・アプローチ

アプリケーションの「ログ出力」をテーマにする。新しい統一されたLoggerインターフェースを使いたいが、既存のレガシーなログライブラリ（異なるメソッド名、引数形式）があるシナリオ。レガシーシステムとの統合というAdapterの典型的な使用シーンを体験できる。

### メリット

- 「ログ出力」はプログラミングで必ず使う実用的なテーマ
- 「レガシーシステムとの統合」はAdapterの最も重要な用途
- コンソール出力からファイル出力への拡張も自然にできる
- 実務でログライブラリを切り替える場面を想定できる

### デメリット

- 「ログ」の概念がプログラミング初心者には馴染みがない可能性
- 複数のログ形式（INFO、ERROR等）を扱うと複雑になりがち
- 地味なテーマのためモチベーション維持が難しい

### 連載構造

| 回 | タイトル | 新しい概念 | ストーリー | コード例1 | コード例2 | 推奨タグ |

| 第2回 | 【Perl/Moo】レガシーロガーとの互換性問題に挑戦 | 異なるインターフェース | 既存のLegacyLoggerが登場。write_log($level, $msg)という異なる形式 | LegacyLoggerクラス定義 | 両方使おうとして困るコード | perl, moo, legacy-code, interface-mismatch, oop-class |

### 差別化ポイント

- **レガシー統合**: Adapterの最も重要な用途である「レガシーシステムとの共存」を体験
- **実務直結**: ログ出力は実際のアプリケーション開発で必ず必要になる
- **拡張性の学習**: 新しいログ出力先（Slack通知等）への拡張も想像しやすい

## 案C: 「データ読み込みツールを作ってみよう」アプローチ

### 特徴・アプローチ

複数のデータ形式（CSV、JSON、固定長テキスト）を統一インターフェースで読み込むツールを作成する。各形式のパーサーが異なるメソッド名や戻り値を持っており、それをAdapterで統一する。データ処理という実用的なテーマで、Adapterパターンの必要性を自然に体感できる。

### メリット

- 「データ読み込み」はPerl入学式卒業者がすぐに使いたいスキル
- CSV、JSONなど具体的なファイル形式で実践的
- 「形式が違うのに同じように扱いたい」という要求が明確
- 前シリーズ（Mooで覚えるオブジェクト指向）との連続性が高い

### デメリット

- JSONパースにはJSON::PPやJSON::XSなどのモジュールが必要
- ファイルI/Oが絡むとエラーハンドリングが複雑になりがち
- データ形式によってはパース処理自体が難しくなる

### 連載構造

| 回 | タイトル | 新しい概念 | ストーリー | コード例1 | コード例2 | 推奨タグ |
|---|---------|-----------|-----------|----------|----------|---------|
| 第1回 | 【Perl/Moo入門】CSVファイルを読み込むクラスの作り方 | 基本クラス設計 | CSVファイルを読み込むCsvReaderクラスを作成 | CsvReaderクラス定義（read_allメソッド） | CSV読み込みスクリプト | perl, moo, csv-parsing, perl-beginner, oop-class |
| 第2回 | 【Perl/Moo】JSONファイルを読み込む別のクラスを追加 | 異なるインターフェース | JsonLoaderが登場。load_json()という異なるメソッド名と戻り値形式 | JsonLoaderクラス定義 | 両方使おうとして困るコード | perl, moo, json-parsing, interface-mismatch, oop-class |
| 第3回 | 【Perl/Moo】異なるデータ形式を統一インターフェースで読む方法 | 委譲とラッピング | JsonLoaderをCsvReader形式で使えるようにするAdapterクラスを作成 | JsonReaderAdapterクラス定義 | Adapterを使った統一的な呼び出し | perl, moo, data-parser, wrapper-class, delegation |
| 第4回 | 【Perl/Moo】CSV・JSON・固定長を同じ方法で処理する設計 | 複数Adapter対応 | 固定長テキスト（FixedWidthParser）も追加してAdapterで統一 | FixedWidthAdapterクラス定義 | 複数形式をループで処理 | perl, moo, multi-format, data-processing, unified-interface |
| 第5回 | 【Perl/Moo】Adapterパターン入門 — GoFデザインパターン解説 | Adapterパターン | ここまで作成したものがAdapterパターンであることを解説、GoFパターンの紹介 | パターン構造図（Mermaid） | 応用例の紹介 | perl, moo, adapter-pattern, gof-patterns, design-patterns |

### 差別化ポイント

- **データ処理特化**: Perl入学式卒業者が最も興味を持つデータ処理を題材に
- **実用性の高さ**: CSVやJSONは実務で頻繁に扱う形式
- **前シリーズとの連続性**: ファイル操作やデータ構造の知識を活用

## 推薦案とその理由

### 推薦：案A「天気情報ツールを作ってみよう」

### 推薦理由

1. **ペルソナとの適合性**
   - 「天気情報」は日常的で親しみやすく、プログラミング初心者でもイメージしやすい
   - 複数サービスを統一するシナリオは、将来的なWeb API利用への橋渡しになる
   - Moo経験者を前提としつつ、新しい概念（インターフェース変換）を段階的に導入

2. **検索意図との適合性**
   - 「Perl API 統合」「Perl 複数サービス」などの検索でヒットしやすい
   - デザインパターン名を前面に出さないため、実用ツールを求める読者も取り込める
   - 「Moo オブジェクト指向」との組み合わせで技術記事としての価値も高い

3. **学習効果**
   - 「同じことをしたいのにAPIが違う」という問題を体験することで、Adapterの必要性を実感
   - 複数のサービスを追加していく過程で、パターンの汎用性を理解できる
   - 最終的にAdapterパターンと明かすことで「なるほど！」という学習効果が得られる

4. **既存シリーズとの連続性**
   - 「Mooで覚えるオブジェクト指向プログラミング」で学んだ委譲（handles）の発展
   - 前シリーズで作成した概念（has、メソッド、コンストラクタ）を自然に活用
   - Facadeパターンシリーズとは異なる「インターフェース変換」の視点を学べる

5. **内部リンクの活用**
   - Mooシリーズ（/2021/10/31/191008/）への参照が自然に入れられる
   - デザインパターン全般の解説記事（Strategy パターン、Facade パターン等）へのリンクも可能
   - 調査ドキュメント（content/warehouse/adapter-pattern.md）の内容を活用可能

### 代替案の選択指針

- **案Aが適しているケース**: API統合に興味がある読者、身近なテーマで学びたい読者
- **案Bが適しているケース**: 実務でログ出力を扱う読者、レガシーシステム統合を学びたい読者
- **案Cが適しているケース**: データ処理に興味がある読者、CSV/JSON操作を学びたい読者
