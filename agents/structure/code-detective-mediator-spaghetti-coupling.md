# コード探偵ロックの事件簿【Mediator】蜘蛛の巣会議〜糸を引く者は誰か〜

## 概要

- **ターゲット層**:
  - 複数モジュールが互いに直接参照し合い、1箇所の変更が全モジュールに波及して保守に苦しんでいる初級〜中級エンジニア。
  - 新しいモジュールを追加するたびに既存の全モジュールを修正しなければならない状況に直面している人。
- **目的**:
  - Mediatorパターンを用いてモジュール間の直接依存をすべて仲介者に集約し、N×N結合をN×1に変換する方法を理解する。
  - 新機能追加時の修正範囲を局所化し、既存コードへの影響をゼロにする設計を体験する。
- **公開予定日**: 2026-03-25 07:07:05 +09:00

## 設定（Phase 1: Prepare）

- **アンチパターン**: Spaghetti Coupling（スパゲッティ結合 / モジュール間のN×N直接依存による密結合）
- **解決パターン**: Mediator パターン
- **依頼人（ワトソン役）**: 森田（アパレルECスタートアップのバックエンドエンジニア・男性・28歳・経験5年）。急成長中のEC「ThreadStyle」の初期メンバーで、バックエンドを一人で書き始めた。ビジネスの成長とともにモジュールが増え、直接参照の蜘蛛の巣に苦しんでいる。一人称は「僕」。
- **舞台**: 社員30人規模のアパレルECスタートアップ。受注・在庫・決済・通知・ログの5モジュールが全方位で直接参照し合い、新機能（クーポンエンジン）追加で全モジュール改修が必要になった。
- **苦境**: 「クーポン機能を追加しろって言われたんですけど、どのファイルを触っても他の全部が壊れるんです。もう、コードのどこにも安全な場所がない」
- **メタファー**:
  - 5モジュールの密結合 = 蜘蛛の巣のように絡み合った糸。1本引けば全体が震える
  - 新モジュール追加 = 蜘蛛の巣にさらに糸を張り巡らす。巣は重みで崩壊寸前
  - Mediator = 巣の中心に座る蜘蛛。すべての振動は中心を通じて伝達される

## 構成案（統合版）

1. **導入：依頼人の訪問**
   - 森田がノートPCを抱えてLCIに駆け込む。モニターには依存関係のグラフ図。
   - ロック：「ほう……見事な蜘蛛の巣だ。だが問題は、蜘蛛が5匹もいることだよ」
   - 森田がクーポン機能追加で5ファイル全てに手を入れる恐怖を語る。

2. **現場検証（Before: アンチパターン）**
   - OrderManager が InventoryManager, PaymentGateway, NotificationService, AuditLogger を直接参照する Before コード。
   - 他のモジュール（PaymentGateway, InventoryManager）も同様に他を直接参照。
   - Mermaid図で10本の依存線が絡み合うグラフを可視化。
   - ロック：「5人の容疑者が全員、互いの秘密を握っている。これでは誰が何を伝えたか追跡不能だ」
   - 初期化コードの地獄（循環参照を避けるための `rw` と後付け設定）。

3. **推理披露（After: Mediatorパターンの導入）**
   - `Colleague` ロール（各モジュール共通インターフェース）の導入。
   - `OrderMediator`（蜘蛛の巣の中心）の実装。
   - 各モジュールは Mediator 経由でのみ通信。`send_event` でイベントを発火。
   - Mermaid図でスター型トポロジーを図示。
   - CouponEngine追加 = Colleague ロールを適用し Mediator に登録するだけ、既存モジュール修正ゼロ。

4. **解決：テストの通過**
   - テスト実行で全パスを確認。
   - ロック：「蜘蛛の巣に中心が生まれた。振動はすべて中心を通じて正しく伝わる」
   - 報酬ジョーク：「依存線の削減率と同じパーセンテージのクーポンをいただこうか」

5. **探偵の調査報告書**
   - 容疑：Spaghetti Coupling（N×N直接依存の密結合）
   - 真実：Mediator パターン
   - 証拠：依存線 10本→5本、新モジュール追加時の既存コード修正ゼロ、影響範囲の局所化
   - 推理のステップ（4項目）
   - ロックより（メッセージ）：Mediatorの本質と注意点（God Mediator化への警告）

## メトリクス比較

| 項目 | 修正前 (Before) | 修正後 (After) | 改善効果 |
|------|-----------------|----------------|----------|
| **モジュール間の依存線** | 10本（N×N直接参照） | 5本（各モジュール→Mediatorのみ） | 50%削減・スター型に整理 |
| **新モジュール追加時の修正** | 既存モジュール複数を修正 | Mediatorにイベント追加のみ | 既存モジュール修正ゼロ |
| **1モジュール変更時の影響範囲** | 直接参照する全モジュール | Mediatorのみ | 影響局所化 |
