# 記事構造案: 探偵は通訳に頼らない（Adapter）

## 1. 基本情報
*   **シリーズ**: コード探偵ロックの事件簿 (code-detective)
*   **テーマ（デザインパターン）**: Adapter
*   **アンチパターン**: 互換性のないインターフェース (Incompatible Interfaces / if文による場当たり的な変換)
*   **予定Slug**: `adapter-pattern`
*   **対象読者**: 古いシステムや外部APIとの連携で、データ変換のif文地獄に陥っている開発者。

## 2. 語り部（ワトソン）のプロファイル
*   **属性**: コピペと最新ツールでその場を凌いできた今どきの新人プログラマ。
*   **背景**: 買収した別会社の古いシステムと、自社の新しいシステムを連携させるタスクを振られた。しかし、データ構造もメソッド名も全く違うため、両者を繋ぐための巨大な「変換用if文だらけのクラス」を作ってしまい、身動きが取れなくなってLCI（探偵事務所）に駆け込んできた。
*   **探偵への態度**: 探偵の古風な言い回しや「ホームズ」要素が全く通じない。「ホームズ？それってAIエージェントのアプリ名ですか？」「『異なる言語を話す者同士の通訳』…？あ、DeepLのAPI繋ぐってことですか？」など、現代的（かつ微妙にズレた）解釈をして探偵のペースを乱す。

## 3. プロット（5幕構成）

### I. 導入：AI通訳と探偵のプライド
*   **状況**: 探偵事務所に、ノートPCを抱えた新人プログラマ（ワトソン）が駆け込んでくる。「違うシステムのAPI同士を繋ぐコードを書いたんですが、仕様変更のたびにif文が増えて、もうわけがわかりません！ChatGPTに直させようとしたら文字数制限で途切れて…」
*   **探偵の反応**: 「AIに頼る前に、美しい設計という名の真実に目を向けたまえ、ワトソン君。初歩的なにおいだよ」
*   **ワトソンのツッコミ**: （ワトソン？誰のことだろ…まあいいか）「はい、じゃあとりあえずこれ見てください！」

### II. 現場検証：互換性なき泥沼（Beforeコード）
*   **コードの症状**: 新しい `UserService` と、古い `LegacyCustomerSystem`。「User」と「Customer」で呼び方が違い、取得メソッドも `get_user_info` と `fetch_customer_data` でバラバラ。ワトソンは、両方に対応するために `if ($type eq 'new') { ... } elsif ($type eq 'legacy') { ... }` という泥臭い分岐コードをごちゃまぜに書いている。
*   **探偵の指摘**: 「言語が違う者同士を、無理やり同じ部屋に押し込めて会話させようとしている。これでは暴動が起きる（バグが出る）のは当然だ」

### III. 推理披露：探偵は通訳（Adapter）を雇う（Afterコード）
*   **リファクタリング**: 探偵が「Adapter（通訳）」パターンを導入する。古いシステムをラップ（包み込む）して、新しいシステムと同じインターフェース（`get_user_info`など）を持つ通訳クラス（Adapter）を作成。
*   **探偵の解説**: 「レガシーな彼に、直接我々の言葉を喋らせるのではない。彼の言葉を我々の言葉に翻訳する『通訳』を間に入れるのだ。それがAdapterパターンだ」
*   **ワトソンの反応**: 「なるほど！ただのラッパークラスですね！最初からそう言ってくださいよ！」（探偵の詩的な表現を台無しにする）

### IV. 解決：シームレスな対話
*   **結果**: 呼び出し側（クライアント）は、相手が新しいシステムか古いシステムかを意識することなく、同じメソッドで通信できるようになった。if文の地獄は消え去った。
*   **結末**: 「見事だ。これで国境はなくなった」とカッコつける探偵に、ワトソンは「ありがとうございます！じゃあ僕、この綺麗になったコードをベースに、別のシステムの分もコピペで作ってみますね！」と駆け出し、探偵が「ちょっと待て！！（Adapterを作る意味がわかってない！）」と引き止めるオチ。

### V. 報告書（まとめ）
*   **解説内容**:
    *   Adapterパターンの目的（既存のクラスを修正せずに、必要なインターフェースに適合させる）。
    *   ラッパーとしての役割。
    *   システム統合やレガシーコード保護における有用性。
