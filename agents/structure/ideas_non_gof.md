# GoF以外のパターンのネタ帳（コード探偵シリーズ用）

## 1. 「Active Record」パターンの限界と「Repository」パターン
* **容疑（アンチパターン）**: 肥大化するモデル (Fat Model)
* **症状**: O/Rマッパーの便利さに甘えて、データアクセスから業務ロジック、バリデーションまで全てを1つのクラス（例: `User` クラス）に詰め込む。結果として数千行の「何でもできる神クラス」となる。
* **真実（解決策）**: Repository（リポジトリ）パターン

## 2. 「どこからでも呼べる便利さ」の代償
* **容疑（アンチパターン）**: Ambient Context（環境コンテキスト）への依存 / 暗黙のグローバル状態
* **症状**: ログインユーザー情報や現在時刻、グローバル設定などを「どこからでも取得できるシングルトンや静的メソッド」から直接取得してしまう。結果、テストのたびにDBやセッションを準備しなければならずテスト困難になる。
* **真実（解決策）**: Dependency Injection（依存性の注入）

## 3. 「とりあえずフラグを足せばいける」の末路（時間がない時の修正あるある）
* **容疑（アンチパターン）**: Magic String / Magic Number（マジックストリング・マジックナンバー）や 複雑な分岐増殖（Arrow Anti-Pattern）
* **症状**: 要件が増えるたびに `if ($status == 1)` や `if ($type === 'premium_user_v2')` のような謎の数値・文字列の条件分岐が無限に増殖し、コードのインデントが深くなっていく。
* **真実（解決策）**: Specification（仕様）パターン

## 4. 「とりあえずデータをまとめる」の罪
* **容疑（アンチパターン）**: 貧弱なドメインモデル（Anemic Domain Model）
* **症状**: `getter/setter`だけのデータを持つDTOクラスばかりになり、本来そのデータで行うべき重要な計算や判断（ロジック）が、それを利用する別の「サービス」クラスに全部吸い上げられてしまう。
* **真実（解決策）**: Tell, Don't Ask（情報をひん剥くな、振る舞いを持たせよ）
