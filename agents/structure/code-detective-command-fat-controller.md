## 2. 物語設定（Phase 0 合意事項）
- **容疑者（アンチパターン）**: Fat Controller / Hardcoded Actions（外部への副作用・状態の変化がコントローラーに直書きされ、取り消し・再実行が不可能な状態）
- **真実（解決パターン）**: Commandパターン（命令をオブジェクトとして封じ込め、実行と取り消しを対にする）
- **ワトソン君（依頼人）**: 外資系決済サービスのAPI連携を担当している若手エンジニア。
    - 状況: 注文エラー時、DBのロールバックは効くが、外部決済APIへの「課金リクエスト」が取り消せず、データの不整合（お代は頂いたが注文はキャンセルされた等）に悩まされている。
- **テーマ**: 外部通信（API）を含む処理において、DBトランザクションだけでは解決できない「取り消し（Undo）」の課題を、Commandパターンを用いて解決すること。

## 3. プロット構成（統合版）

### 導入（[起] 事務所への来客）
- 決済システムの「取り消し」エラーに悩む依頼人（若手エンジニア）がLCI探偵事務所を訪れる。
- 徹夜明けでボロボロのワトソン君に対し、ロックがいつものようにエナジードリンク片手に余裕な態度で迎える。

### 現場検証（[承] 時間の不可逆性）
- 決済処理を担当する `OrderProcessor` クラスを調査。
- `process_order` の中でDBへの保存と、外部決済APIへのHTTPリクエストがベタ書きされている。
- エラーが発生したらDBは `$db->rollback()` で戻せるが、外部APIにはすでに「課金リクエスト」が飛んでしまっているため、取り消すには手動で「返金API」を叩くしかなく、処理が複雑怪奇になっている（Fat Controller化）。

### 推理披露（[転] 実行履歴という名のタイムマシン）
- ロックの解説：「起きてしまった過去（外部通信）はトランザクションでは取り消せない。時間を巻き戻すには、自分が『何をしたか』を正確な記録（オブジェクト）として残しておく必要がある」
- 注文処理（支払いや送信）をただのメソッド呼び出しから、独立した `PaymentCommand` オブジェクトに切り離す。
- `Command::Role` には `execute`（実行）と `unexecute`（取り消し）の2つのメソッドを義務付ける。
- `Invoker`（履歴管理役となるマクロコマンド）を導入し、コマンドスタック（配列）を管理する。エラーが起きたら、スタックから順番に `unexecute` を呼んで巻き戻す。

### 解決（[結] 事件の終わり）
- 決済処理はCommandオブジェクトの送信へと変わり、エラー時には `unexecute` で安全に状態と外部APIがロールバック（返金リクエスト）されるようになる。
- さらにロックからは「履歴がオブジェクトになれば、UndoだけでなくRedo（再実行）や、非同期キューへの保存といった応用も可能だ」と教えられる。
