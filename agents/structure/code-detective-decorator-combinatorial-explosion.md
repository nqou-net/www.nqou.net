# コード探偵ロックの事件簿【Decorator】二十面相のコード〜終わらない継承の連鎖〜

## 概要

- **ターゲット層**:
  - 機能の組み合わせごとにサブクラスを量産してしまい、クラス数が爆発的に増えて管理不能に陥っている初級〜中級エンジニア。
  - 継承ベースの設計しか知らず、「機能を追加＝サブクラスを作る」という思考から抜け出せない人。
- **目的**:
  - Decoratorパターンを用いて、継承による組み合わせ爆発を委譲ベースの動的な機能合成に置き換える手法を理解する。
  - 「必要な機能を必要なだけ重ねる」という柔軟な設計思想を体験する。
- **公開予定日**: 2026-03-23 07:07:05 +09:00

## 設定（Phase 1: Prepare）

- **アンチパターン**: Combinatorial Explosion（サブクラスの組み合わせ爆発）
- **解決パターン**: Decorator パターン
- **依頼人（ワトソン役）**: 佐伯（さえき）。28歳の女性バックエンドエンジニア。経験5年。先輩の退職で通知システムの保守を一人で引き継いだ。責任感が強く、何でも自分で解決しようとする頑張り屋。しかし、サブクラスの増殖に心が折れかけている。一人称は「私」。
- **舞台**: 中堅SaaS企業の社内。顧客向けの通知システム（メール、Slack、ログ記録、リトライ、フィルタリング等）が複雑に絡み合っている。
- **苦境**: 「通知システムの保守を引き継いだんですが、先輩が残していったクラスが30個以上あるんです。`EmailSlackRetryNotifier`、`SlackLogFilterNotifier`、`EmailLogRetryFilterNotifier`……組み合わせの数だけクラスがあって、新しい通知先を1つ追加するだけで既存の組み合わせ分のクラスを全部作り直さなきゃいけなくて。先週、上司に『SMS通知も追加して』と言われた瞬間、目の前が真っ暗になりました」
- **メタファー**:
  - サブクラスの爆発 = 怪人二十面相の変装。犯人が30もの変装で捜査を攪乱しているが、仮面の下の正体は一つ。
  - Decorator = 衣装の重ね着。一枚ずつ必要なコートやマフラーを重ねるだけで、どんな装いにも対応できる着せ替えの技法。
  - ラッピング（委譲） = 証拠品を丁寧に保護袋に包んで保管する探偵の手法。中身はそのまま、外側の包装で機能を追加する。

## 構成案（統合版）

1. **導入：依頼人の訪問**
   - 佐伯がクラス一覧を印刷した紙の束を抱えてLCIに駆け込む。A4用紙にびっしり並んだクラス名の一覧。
   - ロック：「ほう、容疑者リストかね？ ……30人以上とは、なかなか大所帯の犯罪組織だ」
   - 佐伯：「容疑者じゃなくて、全部サブクラスなんです……」
   - ロックが紙を手に取り、鼻を近づけてにおいを嗅ぐ仕草。「この犯人は**変装の名人**だよ、ワトソン君。30の顔を持っているが、正体は一つだ」
   - 佐伯が「ワトソン君」呼ばわりに困惑するが、探偵のペースに巻き込まれる。

2. **現場検証（Before: アンチパターン）**
   - 通知システムのクラス階層を提示。`BaseNotifier` を頂点に、`EmailNotifier`, `SlackNotifier` があり、さらにそこから `EmailRetryNotifier`, `EmailLogNotifier`, `EmailSlackNotifier`, `EmailRetryLogNotifier`... と組み合わせが爆発している。
   - Mermaidクラス図で継承ツリーの異常な広がりを可視化。
   - ロック：「初歩的なにおいだよ、ワトソン君。犯人は**組み合わせ爆発**——継承の仮面を被り、分身を無限に増やす怪人だ」
   - 問題点：N個の機能の組み合わせで最大2^N個のサブクラスが必要、新機能追加で既存クラス数が倍増、各サブクラスでロジックが重複。
   - Beforeコードとして `EmailRetryNotifier` が `EmailNotifier` を継承し、`send` メソッド内でリトライロジックをコピペしている例を示す。

3. **推理披露（After: Decoratorパターンの導入）**
   - ロック：「変装を剥がすのではない。**重ね着**の技法を教えてやるのさ」
   - 共通の `Notifier` ロール（インターフェース）を定義。
   - `EmailSender`, `SlackSender` は基本の通知機能のみ実装。
   - `RetryDecorator`, `LogDecorator`, `FilterDecorator` がそれぞれ `Notifier` ロールを持ち、内部に別の `Notifier` を保持（委譲）。
   - 使用例：`FilterDecorator->new(notifier => RetryDecorator->new(notifier => LogDecorator->new(notifier => EmailSender->new(...))))` で自由に組み合わせ。
   - 佐伯：「30個のクラスが……たった7つに？」
   - ロック：「すべての不吉な継承を排除して残ったものが、いかにシンプルであっても、それが真実（デザインパターン）なんだ」
   - Mermaidクラス図でDecoratorの構造を可視化。

4. **解決：テストの通過**
   - 各Decorator単体のテスト＋組み合わせテストが全て通過。
   - 佐伯が恐る恐る「SMS通知を追加するには……？」と聞く。ロック：「`SmsSender` クラスを1つ作るだけだ。既存のDecoratorはそのまま使える」
   - 佐伯：「1つ……だけ？」と感涙。
   - ロック：「報酬は……そうだな、このクラス階層の深さと同じミリリットル数のバーボンをいただこうか」

5. **探偵の調査報告書**
   - 容疑：Combinatorial Explosion（サブクラスの組み合わせ爆発）
   - 真実：Decorator パターン
   - 証拠：クラス数の劇的削減、機能の自由な組み合わせ、新機能追加時の影響範囲最小化
   - 推理のステップ（4〜5項目）
   - ロックより（佐伯＝ワトソン君へのメッセージ）

## メトリクス比較

| 項目 | 修正前 (Before) | 修正後 (After) | 改善効果 |
|------|-----------------|----------------|----------|
| **クラス数** | 30以上（機能の組み合わせ分） | 基本2 + Decorator 3〜4 = 約7 | 1/4以下に削減 |
| **新機能追加コスト** | 既存組み合わせ×新機能分のクラス新規作成 | Decoratorクラスを1つ追加するだけ | O(2^N) → O(1) |
| **コード重複** | 各サブクラスでリトライ・ログ等のロジックをコピペ | 各Decoratorに1箇所だけ実装 | DRY原則の達成 |
| **テスト容易性** | 組み合わせ分のテストクラスが必要 | 各Decorator単体テスト＋統合テスト | テスト数を大幅削減 |
