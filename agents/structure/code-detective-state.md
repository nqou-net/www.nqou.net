# プロファイルとプロット案

## シリーズ情報
* **シリーズ名**: コード探偵ロックの事件簿
* **テーマ**: State パターン
* **slug**: state
* **タイトル**: コード探偵ロックの事件簿【State】姿なき真犯人〜状態が移り変わる密室〜

## 語り部（ワトソン役）プロファイル
* **属性**: 20代後半の若手バックエンドエンジニア。真面目だが疲弊している。
* **背景**: 中規模ECサイトの注文ステータス管理システムを引き継いだ。未払い、支払い済み、出荷準備中、発送済み、キャンセルなどの複雑な状態遷移がすべて1つの巨大な `if-elsif` 文で制御されており、新機能の追加（例えば「返品処理中」ステータス）のたびにどこかが壊れる「スパゲッティ状態遷移（Spaghetti State）」に苦しんでいる。
* **一人称**: 私

## プロット構成（ミステリー5幕構成）

### I. 導入（依頼）: 事務所への来客
依頼人（私）が、変更するたびに壊れるECサイトのステータス管理コードを手に探偵事務所を訪れる。室内は排熱で暑く、探偵ロックは長さをミリ数に換算したバーボンではなく、今回は強烈なエナジードリンクを飲んでいる。「初歩的な匂いだよ、ワトソン君」と勝手に助手に任命される。

### II. 現場検証: コードの指紋、容疑者の浮上
探偵とともに問題の `Order` クラス（Beforeコード）を検証する。そこには `process_payment`, `ship_item`, `cancel` などの各メソッドの中に、`$self->status` の値によって振る舞いを変えるための長大な `if-elsif` ブロックが鎮座していた。探偵は「これが真犯人。状態の変化という密室の中で、無数のフラグが暴走しているんだ」と指摘する。

### III. 推理披露: 鮮やかなリファクタリング、真実の開示
探偵は推理の切り札「State パターン」を提示する。巨大な `if` 構文を解体し、各状態（`Unpaid`, `Paid`, `Shipped` など）を独立した状態クラスとして切り出す。`Order` クラスは現在の状態オブジェクトを保持し、操作を委譲するだけになる。条件分岐が消え去り、各状態クラスが自律的に振る舞いと状態遷移の責任を持つ構造（Afterコード）が明かされる。私はその鮮やかな手口に驚愕する。

### IV. 解決: 事件の終わり、平和なビルド
新しく「キャンセル」状態の振る舞いを追加しようとした場合も、既存のクラスを書き換えることなく、新しい状態クラスを追加するだけで済むことを確認する。テスト（ビルド）が緑色に点灯し、システムに平和が訪れる。

### V. 報告書: 探偵の調査報告書
探偵がまとめた調査報告書。アンチパターン（スパゲッティ状態遷移）と解決策（Stateパターン）、そして推理のステップ（リファクタリング手順）をまとめ、ロックからワトソン君へのキザなメッセージで締めくくる。
