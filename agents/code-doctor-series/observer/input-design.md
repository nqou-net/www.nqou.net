# Input Design: Observer

## 選定結果

| 要素 | 採用案 | スコア |
|------|--------|--------|
| テーマ | 推し活ゲリラライブ通知Bot（逆転） | 22/25 |
| 患者 | 歴15年のインフラエンジニア（革新） | 22/25 |
| 症状 | 激しすぎるポーリング過呼吸（革新） | 22/25 |

## テーマ詳細
- **テーマ名**: 地下アイドルのゲリラライブ・グッズ販売通知Bot
- **一行説明**: 推しのアイドルの突発的なイベントやグッズ販売開始を最速で検知・通知するシステム。
- **パターンとの接点**: 情報元（Subject: アイドルのサイト/SNS）の状態更新を、複数のチャネル（Observer: LINE, Discordアカウント連携されたファン各位）に即座にブロードキャストする必要があり、Observerパターンの絶好の適用例となる。
- **読者像**: 推し活の熱量に共感する層。または、イベント駆動なアーキテクチャへの移行に興味がある中堅エンジニア。

## 患者詳細
- **属性**: 40代、インフラ・バックエンドエンジニア、歴15年。
- **性格キーワード**: カタブツ、Cron絶対主義、最近地下アイドルにドハマりした
- **背景**: サーバー監視とCron運用で生きてきた堅物。最近、ふとしたきっかけで地下アイドル「ピュア・バイナリ」にハマり、ゲリラ告知されるライブチケットを確保すべく、自らのインフラ技術（Perlスクリプト）を全投入してシステムを構築した。
- **主訴**: 「推しのサイトが落ちるんです。私の通知システムのせいかもしれません……愛によるDDoS攻撃です」
- **コメディポテンシャル**: 「1秒ごとのCronジョブ」「異常なトラフィック」という無骨な言葉と、「推し」「グッズ最速入手」というオタク用語の極端なギャップ。愛が重すぎて相手のサーバーを落としかける暴走。

## 症状詳細
- **技術的症状**: 状態変化を「検知（Pull）」するために、対象サイトに対し1秒間隔の無限ループ/無限CronでHTTPリクエストを浴びせ続ける力技。通知先が増えるたびにポーリング処理の中に`curl`の呼び出しがハードコードで増殖している。
- **医療メタファー**: 「ポーリング過呼吸（Hyper-Polling Syndrome）」または「執着性DDoS型・愛着障害」。
- **重症度**: 重症（相手のサーバーに実害が出かねないため緊急オペが必要）
- **Before/Afterの見込み**: Before: 力技による定周期Pull型アーキテクチャ。After: Subjectの状態変化時に`notify()`をトリガーするPush型のObserverパターンへのリファクタリング。
- **読者の「あっ」ポイント**: 「状態変化を知りたい」ときに、つい自分から聞きに行ってしまう（Pull型）設計の脆さに気づき、相手から教えてもらう（Push型）というパラダイムシフトを直感できる点。

### 3要素の整合性
- テーマ × 患者: ✅ カタブツなおじさんがアイドルにハマってインフラスキルを悪用（？）する構図が極めて滑稽で魅力的。
- テーマ × 症状: ✅ 「1秒でも早く知りたい」というオタクの心理と、過剰なポーリングが技術・感情の両面で完全一致。
- 患者 × 症状: ✅ インフラ一筋だからこそ、アプリ層の設定（Webhookなど）より、外形監視的な力技のCronポーリングに走るのが圧倒的にリアル。

## 不採用案（参考）
- **テーマ案A（王道）**: ECサイトの在庫・価格変動通知システム（スコア 18） - 実用的だが、既存の記事やよくある解説と差別化が難しい。
- **テーマ案B（革新）**: スマート農業の土壌・気象センサー網（スコア 16） - ニッチすぎて読者の身近な共感を得にくい。
- **患者案A（王道）**: 入社2年目のフロントエンドエンジニア（スコア 17） - キャラとしては弱く、物語のうねりが作りにくい。
- **患者案C（逆転）**: 元パティシエの異業種エンジニア（スコア 17） - 属性が渋滞しそう。
- **症状案A（王道）**: ハードコードの密結合（スコア 19） - 典型例としては良いが、コメディとしてのパンチに欠ける。
- **症状案C（逆転）**: 逆流した依存関係（スコア 20） - 少し難解で、初心者が直感的に笑えない危険性。
