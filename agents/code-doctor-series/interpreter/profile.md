# Medical Chart

## 症状 (Symptoms)

### 技術的症状
- 管理画面から入力されたルール文字列（例: `$cart_total >= 5000 && $member_rank eq "gold"`）を Perl の `eval` で直接評価
- ルール文字列の前処理は正規表現による置換のみ。括弧のネストや演算子の優先順位は eval に丸投げ
- eval 内でエラーが発生した場合は `$@` を握りつぶして「条件不一致」として処理
- ルール数は 40 を超え、デバッグ時にどのルールがマッチしたか追跡困難
- セキュリティ監査で「コードインジェクション: Critical」と判定

### 医療メタファー
「**急性eval中毒**」

`eval()` という強力だが危険な薬物に依存し、離脱できない状態。即効性があり「効く」ため本人は問題を自覚していない。しかし薬物（eval）は全身（システム全体）を蝕んでおり、外部からの汚染物質（悪意のある入力）をフィルタリングせずに直接体内（実行環境）に取り込むリスクを抱えている。さらに副作用（エラーの握りつぶし）により、異変の早期発見も不可能になっている。

## 診断 (Diagnosis)

**病名**: 急性eval中毒（Acute eval Intoxication）  
**原因**: 動的評価の安易な手段として eval に依存。条件式を「言語構造」ではなく「文字列」として扱い続けた  
**合併症**: 
- コードインジェクション感受性（外部入力が直接実行環境に到達）
- エラーハンドリング麻痺（`$@` の握りつぶしにより異常検知不能）
- デバッグ不能症（40超のルール文字列のどれがマッチしたか追跡不可）

## 処方 (Prescription)

**治療法**: Interpreter Pattern 適用手術  
**手術概要**:
1. ルール文字列を「トークン」に分解するレキサーの実装（血液浄化）
2. トークン列を Expression オブジェクトのツリー（AST）に変換するパーサーの実装（臓器再建）
3. 各 Expression クラスに `interpret($context)` メソッドを持たせ、再帰的に安全に評価（免疫系の再構築）
4. eval の完全除去（毒物の排出）

**Expression クラス階層**:
- `Expression`（基底） → `interpret($context)`
- `ComparisonExpression`（終端式）: `>=`, `==`, `eq` など
- `AndExpression`（非終端式）: 左辺 AND 右辺
- `OrExpression`（非終端式）: 左辺 OR 右辺
- `NotExpression`（非終端式）: 否定
- `LiteralExpression`（終端式）: 数値・文字列リテラル
- `VariableExpression`（終端式）: コンテキスト変数参照

**予後**:
- eval 完全除去によりコードインジェクションリスク消滅
- 新しい演算子の追加は新 Expression クラスの追加のみ
- 各ルールの評価過程がオブジェクトツリーとして可視化可能（デバッグ容易に）
- ルール文字列のパース時にバリデーション可能（不正な式を早期検出）

## 来院 / 往診

**来院**。セキュリティ監査の指摘を受け、知人のエンジニアから「コード診療所」を紹介される。CTOの「1ヶ月以内に対応」という指令に焦りつつも、「eval を剥がしたら全部壊れる」という恐怖から自力での対処を諦め、コード診療所を訪れる。ただし来院時も「eval のどこがダメなんすか」と強気。
