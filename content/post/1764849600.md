## エージェントパネルの画面構成

エージェントパネルは、効率的な作業のために最適化された4つの主要エリアで構成されています。

### 1. タスク入力エリア（Task Input Area）

画面上部に配置され、実行したいタスクを記述する場所です。

**基本構成：**
```
┌─────────────────────────────────────────────┐
│ タスクを入力してください                     │
│                                              │
│ [テキストエリア - 複数行入力可能]            │
│                                              │
│ [実行] [クリア] [テンプレート▼]             │
└─────────────────────────────────────────────┘
```

**機能：**
- マルチライン入力対応
- Markdown 形式でのフォーマット
- 過去のタスク履歴からの再利用
- テンプレート機能

**ショートカットキー：**
- `Ctrl/Cmd + Enter`: タスク実行
- `Ctrl/Cmd + K`: 入力クリア
- `↑/↓`: 履歴の参照

### 2. エージェント選択エリア（Agent Selection Area）

タスクを実行するエージェントを選択・設定するエリアです。

**表示内容：**
```
利用可能なエージェント：
  ☑ task-planner        (タスク計画)
  ☑ implementation      (実装)
  ☑ code-reviewer       (レビュー)
  ☐ documentation       (ドキュメント)
  ☐ testing             (テスト)
  
[カスタムエージェントを追加...]
```

**エージェントの種類：**

| エージェント名 | 役割 | 推奨用途 |
|--------------|------|---------|
| task-planner | タスク分解・計画立案 | 複雑な機能開発の初期段階 |
| implementation | コード実装 | 新機能の実装、リファクタリング |
| code-reviewer | コードレビュー | 品質確保、バグ検出 |
| documentation | ドキュメント生成 | README、API ドキュメント |
| testing | テスト生成 | ユニットテスト、E2Eテスト |
| security-checker | セキュリティ検証 | 脆弱性検出 |

### 3. 実行ステータスエリア（Execution Status Area）

各エージェントの実行状態をリアルタイムで表示します。

**表示例：**
```
実行中のタスク: "ユーザー認証機能の実装"

[task-planner] ✓ 完了 (2.3秒)
  └─ タスクを5つのサブタスクに分解しました

[implementation] ⚙ 実行中... (15秒経過)
  ├─ auth.js を生成中... ✓
  ├─ middleware.js を生成中... ⚙
  └─ config.js 待機中...

[code-reviewer] ⏳ 待機中
  └─ implementation の完了を待っています
```

**ステータスアイコン：**
- ✓: 完了
- ⚙: 実行中
- ⏳: 待機中
- ⚠: 警告あり
- ✗: エラー

### 4. 結果・ログ表示エリア（Results & Logs Area）

実行結果の詳細とログを表示します。

**タブ構成：**
```
[概要] [詳細ログ] [生成ファイル] [提案事項]

─── 概要タブ ───
✓ タスク完了
  実行時間: 45.2秒
  生成ファイル: 3個
  警告: 0個
  
主な成果物:
  - src/auth/auth.js (新規作成)
  - src/middleware/auth-middleware.js (新規作成)
  - config/auth-config.js (新規作成)
```

## 効果的なタスク入力のコツ

### 1. 明確な目的を記述する

**悪い例：**
```
認証機能を作って
```

**良い例：**
```
Node.js + Express で JWT を使用したユーザー認証機能を実装してください。

要件：
- メールアドレスとパスワードでのログイン
- JWT トークンの発行と検証
- パスワードは bcrypt でハッシュ化
- ミドルウェアで保護されたルートの作成

技術スタック：
- Node.js 18.x
- Express 4.x
- jsonwebtoken
- bcrypt
```

### 2. コンテキストを提供する

既存のプロジェクト構造やコーディング規約を明示します：

```
プロジェクト構造：
src/
  ├─ controllers/
  ├─ models/
  ├─ routes/
  └─ middleware/

コーディング規約：
- ES6+ モジュール構文を使用
- async/await でエラーハンドリング
- JSDoc コメントを記載
```

### 3. 期待する成果物を具体的に

```
期待する成果物：
1. 認証ロジックの実装ファイル
2. Express ミドルウェア
3. ユニットテストコード
4. API ドキュメント（Markdown形式）
```

## ワークフローパターン

### パターン1: 段階的実装

小さなタスクを順番に実行していくアプローチです。

```
ステップ1: 計画
  タスク: "ユーザー認証機能の実装計画を立ててください"
  エージェント: task-planner のみ

↓ 結果を確認

ステップ2: 実装
  タスク: "計画に基づいて認証ロジックを実装してください"
  エージェント: implementation のみ

↓ 結果を確認

ステップ3: レビュー
  タスク: "実装したコードをレビューしてください"
  エージェント: code-reviewer のみ
```

**メリット：**
- 各段階で結果を確認できる
- 問題の早期発見
- 学習効果が高い

**適用場面：**
- 初めて使う機能
- 重要度の高いタスク
- チームで作業内容を共有したい場合

### パターン2: 自動オーケストレーション

すべてのエージェントを有効にして、自動的に連携させるアプローチです。

```
タスク: "ユーザー認証機能を実装してください（計画からテストまで）"

エージェント: すべて有効
  ☑ task-planner
  ☑ implementation
  ☑ code-reviewer
  ☑ testing
  ☑ documentation

→ エージェントが自動的に協調動作
```

**メリット：**
- 最速での完了
- 一貫性のある成果物
- 手間が少ない

**適用場面：**
- 定型的なタスク
- 時間が限られている場合
- エージェントの設定が十分に調整されている場合

### パターン3: 反復改善

レビューと修正を繰り返すアプローチです。

```
サイクル1:
  実装 → レビュー → 問題検出

サイクル2:
  修正 → レビュー → さらなる改善点検出

サイクル3:
  最終調整 → レビュー → 承認
```

**実行例：**
```
# 初回実装
タスク: "ユーザー登録APIを実装"
エージェント: implementation, code-reviewer

# レビュー結果を見て修正
タスク: "レビューで指摘されたエラーハンドリングを改善"
エージェント: implementation, code-reviewer

# 最終確認
タスク: "セキュリティの観点で再レビュー"
エージェント: code-reviewer, security-checker
```

## エージェントの設定とカスタマイズ

### 基本設定の変更

各エージェントの動作は設定パネルで調整できます：

```
[implementation エージェント設定]

出力スタイル:
  ◉ 詳細（コメント多め）
  ○ 標準
  ○ 簡潔（コメント少なめ）

コーディング規約:
  ☑ ESLint 規約に従う
  ☑ Prettier でフォーマット
  ☐ TypeScript 型定義を生成

ファイル生成:
  ☑ 既存ファイルのバックアップを作成
  ☑ Git でコミット前に確認
```

### 優先度の設定

複数のエージェントが矛盾する提案をした場合の優先順位：

```
優先度設定:
  1. security-checker (最優先)
  2. code-reviewer
  3. implementation
  4. documentation
```

## 実践演習：ブログ記事生成ワークフロー

実際に Hugo ブログの記事生成ワークフローを構築してみましょう。

### ステップ1: タスク定義

```
タスク: Hugo ブログ記事の生成

要件:
- タイトル: "GitHub Actions で CI/CD を構築する"
- カテゴリ: DevOps
- タグ: github-actions, ci-cd, automation
- 文字数: 約2000字
- コードスニペット: 3つ以上
- front matter は YAML 形式

記事構成:
1. はじめに
2. GitHub Actions の基本
3. CI/CD パイプラインの設定
4. 実践例
5. まとめ
```

### ステップ2: エージェント選択

```
☑ task-planner        記事の構成を計画
☑ implementation      記事内容を生成
☑ code-reviewer       コードスニペットの検証
☑ documentation       front matter とメタデータ
```

### ステップ3: 実行と確認

実行ボタンをクリック後、以下のような流れで処理されます：

```
[00:00] task-planner 開始
[00:02] ✓ 記事を6セクションに分解
        1. はじめに（200字）
        2. GitHub Actions 概要（400字）
        3. ワークフロー基本（500字）
        4. CI 実装例（500字）
        5. CD 実装例（500字）
        6. まとめ（300字）

[00:03] implementation 開始
[00:05] ✓ front matter 生成完了
[00:12] ✓ 本文セクション1-3完了
[00:18] ✓ 本文セクション4-6完了

[00:19] code-reviewer 開始
[00:21] ✓ YAML 構文チェック: OK
        ⚠ コード例1: インデント修正推奨
        ✓ コード例2-3: OK

[00:22] implementation (修正)
[00:23] ✓ インデント修正完了

[00:24] 完了！
        生成ファイル: content/post/github-actions-cicd.md
```

### ステップ4: 結果の確認

生成されたファイルを確認：

## 実行結果の解釈と活用

### ログの読み方

詳細ログタブで、エージェントの思考プロセスを確認できます：

```
[task-planner - 思考プロセス]
分析: タスクは「ブログ記事生成」
  → 必要な要素: タイトル、本文、メタデータ
  → 技術要件: Hugo, Markdown, YAML

計画立案:
  1. メタデータ生成（front matter）
  2. 記事構成の決定
  3. 各セクションの執筆
  4. コードスニペットの挿入
  5. 最終レビュー

推定時間: 約20秒
```

### 警告への対処

警告が出た場合の対処法：

```
⚠ 警告: コードスニペットのインデントが不統一

推奨アクション:
  1. 自動修正を適用 [適用]
  2. 手動で修正 [エディタで開く]
  3. 無視 [スキップ]
```

### 提案事項の活用

エージェントからの改善提案を確認：

```
[提案事項]

📝 品質向上のための提案:
  - 記事の冒頭に目次を追加すると読みやすくなります
  - 図解を1-2個追加すると理解が深まります
  - 関連記事へのリンクを追加できます

🔧 技術的な提案:
  - front matter に description を追加すると SEO 効果が上がります
  - OGP 画像を設定すると SNS シェア時の見栄えが良くなります
```

## トラブルシューティング

### よくある問題と解決策

**問題1: エージェントが途中で停止する**

```
原因: タイムアウト、リソース不足
解決策:
  - タスクをより小さな単位に分割
  - 不要なエージェントを無効化
  - 実行優先度を調整
```

**問題2: 期待と異なる結果が生成される**

```
原因: タスク記述が曖昧
解決策:
  - より具体的な要件を記述
  - 例を提示する
  - 段階的アプローチに変更
```

**問題3: エージェント間で矛盾した提案**

```
原因: 優先度設定が不適切
解決策:
  - エージェント優先度を見直す
  - 矛盾する項目を明示的に指定
  - 段階的実行に切り替え
```
