「ロックさん！ 俺の書いた素晴らしい決済ロジックが、ただの『通知屋さん』に成り下がってしまったんです！」

探偵事務所のドアを勢いよく開けたのは、若手バックエンドエンジニアのマイクだった。
彼の服（コード）には、至るところに強烈なニオイを放つベタベタの接着剤がこびりついている。

「落ち着け、マイク」
私、コード探偵のロックは、愛用のパイプ（実は単なる電子タバコだ）をふかしながら彼をなだめた。「見事なほど接着剤まみれだな。何と何がくっついてしまったんだ？」

「決済完了のシステムです」マイクは泣き出しそうな顔でテーブルに大量の印刷物を広げた。「最初は『決済完了時にメールを送る』だけだったんです。でも、SMS対応しろ、Slackにも流せ、次はPush通知だ、と要求が絶え間なく続いて……」

### 今回の容疑者：Hardcoded Instantiation（newの呪縛）

| プロファイル | 内容 |
|:---|:---|
| **容疑者（アンチパターン）** | Hardcoded Instantiation（ハードコードされたインスタンス化） |
| **罪状** | メインロジックの中に特定の具象クラスの `new` を直接書き込み、条件分岐を肥大化させる罪 |
| **解決策（デザインパターン）** | Factory Method パターン |
| **対象読者** | IF文でオブジェクトを生成し分けている人、新しいクラスを追加する際に既存コードを修正している人 |

### 事件現場のコード（Before）

マイクの広げたコードベースの中核、`OrderService.pm` には以下のような凄惨な現場が残されていた。

```perl
# --- 通知機能の各クラス ---
package EmailNotifier {
    use Moo;
    sub send ($self, $message) { return "Email sent: $message"; }
}

package SmsNotifier {
    use Moo;
    sub send ($self, $message) { return "SMS sent: $message"; }
}

package SlackNotifier {
    use Moo;
    sub send ($self, $message) { return "Slack message sent: $message"; }
}

# --- メインロジック（問題があるクラス） ---
package OrderService {
    use Moo;
    use Carp qw(croak);

    sub complete_order ($self, $order_id, $notification_type) {
        my $message = "Order $order_id has been completed.";
        my $notifier;

        # ！！強烈な接着剤（new）の現場！！
        if ($notification_type eq 'email') {
            $notifier = EmailNotifier->new();
        } elsif ($notification_type eq 'sms') {
            $notifier = SmsNotifier->new();
        } elsif ($notification_type eq 'slack') {
            $notifier = SlackNotifier->new();
        } else {
            croak "Unknown notification type: $notification_type";
        }

        # 通知の実行
        my $result = $notifier->send($message);
        return $result;
    }
}
```

「美しいだろう？」マイクが大真面目に言った。
「動くことには動くからな」私はため息をついた。「だか、マイク。君は大きな勘違いをしている。**『new』はただのインスタンス生成演算子じゃない。システム上で使える最も強力な『接着剤』だ。**」

「接着剤？」

「そうだ。`OrderService` は、注文を完了させるのが本来の仕事のはずだ。なのに今のこいつは、**『どの通知クラスをどうやって生成するか（newするか）』という知識まで抱え込んでいる。**」

私はコードの `if-elsif` ブロックを指差した。
「通知の種類が増えるたびに、君はこの決済の心臓部である `OrderService` のお腹を切り開いて、新しい `elsif` をねじ込んでいる。これではいつか、心臓（決済ロジック）を傷つけてバグを生む（Open-Closed Principle違反）。」

「じゃあ、ど、どうすれば……。通知しないわけにはいかないんですよ」

「簡単さ。**『使う責任』と『作る責任』を切り離せばいい。**」

私はホワイトボードに向かい、一つの工場（Factory）の絵を描いた。

### 推理と解法：Factory Methodの導入

「ピザ屋を想像してくれ」私は説明を始めた。「客は『マルゲリータをくれ』と注文して、出てきたピザを食べる。これが**『使う側（OrderService）』**だ」

マイクはうなずいた。

「客がいちいち厨房に入って、小麦粉から生地をこねて、オーブンの温度を調整して焼き上げたりしないだろう？ それは**『作る側（Factory）』**の仕事だ。君の `OrderService` は、ピザの生地まで自分でこねているようなものなんだよ」

#### 鮮やかな切り離し（After）

私は彼のコードからべったりと張り付いた `new` の塊を切り剥がし、新しいファイル `NotifierFactory.pm` に移した。

```perl
# --- Factory クラス ---
package NotifierFactory {
    use Moo;
    use Carp qw(croak);

    # 生成ロジック（どの種類をどう生成するか）をここに閉じ込める
    sub create ($self, $type) {
        if ($type eq 'email') {
            return EmailNotifier->new();
        } elsif ($type eq 'sms') {
            return SmsNotifier->new();
        } elsif ($type eq 'slack') {
            return SlackNotifier->new();
        } else {
            croak "Unknown notification type: $type";
        }
    }
}
```

「そして、肝心の `OrderService` のお腹の中はこうなる」

```perl
# --- メインロジック（改善されたクラス） ---
package OrderService {
    use Moo;

    # 工場（Factory）を外部から受け取る
    has 'notifier_factory' => (
        is => 'ro',
        required => 1,
    );

    sub complete_order ($self, $order_id, $notification_type) {
        my $message = "Order $order_id has been completed.";

        # 工場に「このタイプの通知器を作ってくれ！」と頼むだけ
        my $notifier = $self->notifier_factory->create($notification_type);

        my $result = $notifier->send($message);
        return $result;
    }
}
```

マイクは目を見開いた。「`OrderService` から、`EmailNotifier` や `SlackNotifier` の名前が完全に消え去った……！」

「そうだ。`OrderService` が知っているのは、『工場に `create` と頼めば、自分に代わって正しい通知オブジェクト（Notifier）を作ってくれる』ことだけだ。そして受け取ったオブジェクトに対して `send` を呼ぶ（**使う**）ことに専念できる」

```perl
# 実行する側（例えばコントローラーやバッチスクリプト）
my $factory = NotifierFactory->new();
my $service = OrderService->new( notifier_factory => $factory );

# 利用
$service->complete_order('12345', 'email');
```

「これなら！」マイクは膝を叩いた。「明日、営業部から『次はLINE通知を追加してくれ』って言われても、`OrderService` は一切変更しなくていい！ `NotifierFactory` の中に `line` 分岐を一つ足すだけで済むんですね！」

（※注：より高度な構成では、Factoryの中の `if` すらなくすために、設定ファイルや文字列から動的にモジュールを `require` する『Abstract Factory』的なアプローチに進むこともできるが、今回は基礎としての分離に留めている）

### 結末：テスト用の「モック工場」

「ああ。しかも、この分離にはもう一つ決定的なメリットがある」

私は最後に、テストコードの断片を彼に見せた。

```perl
# テスト専用の「偽物の工場」を作る
package MockFactory {
    use Moo;
    sub create ($self, $type) {
        return MockNotifier->new(); # 絶対に外に通信しない偽物
    }
}

# 決済完了テストを呼び出すときだけ、偽物を注入する
my $mock_factory = MockFactory->new();
my $test_service = OrderService->new( notifier_factory => $mock_factory );

$test_service->complete_order('999', 'sms'); 
```

「『作る責任』を外部の Factory に完全に委譲したからこそ、テストの時だけ『絶対にメールやSMSを誤送信しない、ダミーの工場（MockFactory）』にすり替えることができるんだよ」

「……完璧です、ロックさん」
マイクの顔から、さっきまでの絶望感は消え去っていた。「これなら他の開発者が触っても壊れない。僕はこれで、決済処理という本業に堂々と集中できます！」

マイクは軽くなった足取りで事務所を後にして行った。

私は冷めたコーヒーをすすりながら、パソコンのディスプレイを見つめた。
世の中のコードは、今日も至る所で不要な `new` によって固く結びつけられている。接着剤を解きほぐす私の仕事は、まだまだ尽きそうにない。

## 探偵の調査報告書

**【Factory Method パターンの要点】**

- **定義**: インスタンスの生成をサブクラス（または専用のFactoryクラス）に委譲するパターン。
- **目的**: オブジェクトの「使用」と「生成」の責任を分離し、結合度を下げること。
- **今回の治療効果**:
  - `OrderService` から具象クラスへの依存（`EmailNotifier`等の名前）を排除した。
  - 新機能追加時も中核ロジックの修正が不要となり、**Open-Closed Principle（変更には閉じ、拡張には開かれているべき）** を満たした。
  - 依存性の注入（DI）と組み合わせることで、ユニットテストが飛躍的に書きやすくなった。

「生成」のコードが複雑になり始めたり、複数の場所で同じ `new` と `if` の羅列が現れたら、それは**専門の工場**を建てるべきタイミングだということを忘れないようにしよう。
