「CSVインポートの処理を直すたびに、別の処理が壊れるんです」

私は佐々木遥、二十六歳。社内の売上集計ツールを一人で開発しているエンジニアだ。

このツールの中心にあるのが、CSVインポート処理。各店舗から送られてくる売上CSVを読み込み、集計して月次レポートを作る。たったそれだけの処理なのに、なぜかバグが絶えない。

原因はわかっていた。インポート処理が、一つの巨大なメソッドになっているのだ。空行を飛ばし、ヘッダーをスキップし、カラムを分割し、トリムし、バリデーションし、合計を計算する。全部が一つのメソッドに詰め込まれている。

先週、合計計算のロジックを直したら、バリデーションが壊れた。同じメソッドの中でループ変数を共有していたのだ。修正のたびにメソッド全体を読み解く必要がある。三十分の修正に三時間の読解が伴う。

「レガシー・コード・インベスティゲーション（LCI）」

雑居ビルの三階。扉を開けると、エナジードリンクの缶が一列に並んでいた。七本。それぞれの缶にポストイットが貼られている。「除去」「分割」「検証」——どうやら缶に名前をつけているらしい。（探偵事務所なのか、小学生の自由研究なのかわからない）

「——ほう。全員同房の事件だね」

「佐々木です。全員同房って何ですか」

「一つの部屋に全員押し込めば管理が楽だと思うだろう？　だが実際には、全員が互いの足を踏み合うだけだ。コードも同じだよ、ワトソン君」

「あの、佐々木です。……どういう意味ですか？」

「証拠を見せたまえ。犯人の数を数えよう」

## 現場検証：全員同房の容疑者たち

コードを見せると、ロックは `import_csv` メソッドを食い入るように読み始めた。

```perl
package CsvImporter;
use Moo;

sub import_csv {
    my ($self, $lines) = @_;

    my @results;
    my $is_header = 1;
    for my $line (@$lines) {
        # 空行スキップ
        next if $line =~ /^\s*$/;

        # ヘッダースキップ
        if ($is_header) {
            $is_header = 0;
            next;
        }

        # カラム分割
        my @cols = split /,/, $line;

        # バリデーション: カラム数チェック
        next unless @cols == 3;

        # カラム変換: 名前のトリム、金額の数値化
        my $name   = $cols[0];
        $name =~ s/^\s+|\s+$//g;
        my $amount = $cols[1];
        $amount =~ s/[^0-9]//g;
        my $date   = $cols[2];
        $date =~ s/^\s+|\s+$//g;

        # バリデーション: 金額が正の数
        next unless $amount > 0;

        push @results, { name => $name, amount => int($amount), date => $date };
    }

    # 合計計算
    my $total = 0;
    $total += $_->{amount} for @results;

    return { records => \@results, total => $total };
}
```

ロックは指を折りながら数え始めた。

「一人目、空行除去。二人目、ヘッダースキップ。三人目、カラム分割。四人目、カラム数バリデーション。五人目、フィールド変換。六人目、金額バリデーション。七人目、合計計算」

「七つの処理が一つのメソッドにあるって、そういう数え方ですか」

「七人の容疑者が全員同じ房に入れられている。一人が動けば隣の人間にぶつかる。合計計算を直そうとして、バリデーションの変数に触れてしまう。心当たりがあるだろう？」

私は先週のバグを思い出した。ロックの言う通りだった。

「初歩的なにおいだよ、ワトソン君。**God Method**——一つのメソッドにすべてを詰め込む。容疑者を全員同房にしておけば監視が楽だと思ったら、全員が共犯者になるだけだ」

「でも、全部まとめて一つのメソッドにした方が、流れが見やすいんじゃ……」

「流れが見やすい？　このメソッドは何行ある？」

「……四十行くらいです」（本番コードはもっと長い。百行を超えている）

「四十行の中に七つの責務が混在している。空行除去だけをテストしたいとき、どうする？」

「……このメソッド全体を呼ぶしかありません」

「つまり、一人の容疑者を取り調べるために、七人全員を取調室に入れなければならない。初歩的な問題だよ」

## 推理披露：容疑者を個別の取調室へ

「解決策は、容疑者を一人ずつ個別の取調室に分離することだ。**Pipeline パターン**——処理をフィルターのチェーンとして構成する」

「パイプライン？　工場のライン作業みたいなものですか？」

「まさにその通り。データが一つのベルトコンベアの上を流れ、各工程で一つずつ処理される。各工程は自分の仕事だけを知っていればいい」

ロックはまず、パイプライン本体を書いた。

```perl
package CsvPipeline;
use Moo;

has filters => (is => 'ro', required => 1);

sub execute {
    my ($self, $data) = @_;
    my $result = $data;
    for my $filter (@{ $self->filters }) {
        $result = $filter->process($result);
    }
    return $result;
}
```

「パイプライン自体はこれだけだ。`filters` にフィルターの配列を渡し、`execute` でデータを順番に流す。各フィルターは `process` メソッドを持ち、受け取ったデータを加工して次に渡す」

「これだけ？　ずいぶんシンプルですね」

「パイプラインの仕事は、データを流すことだけだ。加工するのはフィルターの仕事。責務を分けたまえ」（偉そうに言うけど、それを分けること自体が難しかったんだ）

ロックは続けてフィルターを書き始めた。

```perl
package Filter::SkipBlankLines;
use Moo;
sub process {
    my ($self, $lines) = @_;
    return [ grep { $_ !~ /^\s*$/ } @$lines ];
}
```

「一人目の容疑者、空行除去。自分の取調室を持ち、空行を消すことだけに集中する」

```perl
package Filter::SkipHeader;
use Moo;
sub process {
    my ($self, $lines) = @_;
    return [ @{$lines}[1 .. $#$lines] ];
}
```

「二人目、ヘッダースキップ。先頭行を飛ばすだけだ」

```perl
package Filter::ParseColumns;
use Moo;
sub process {
    my ($self, $lines) = @_;
    return [ map { [ split /,/, $_ ] } @$lines ];
}
```

「三人目、カラム分割。文字列を配列に変える」

```perl
package Filter::ValidateColumnCount;
use Moo;
has expected => (is => 'ro', default => 3);
sub process {
    my ($self, $rows) = @_;
    return [ grep { scalar @$_ == $self->expected } @$rows ];
}
```

「四人目、カラム数バリデーション。期待するカラム数を `expected` で設定できる。三列のCSVなら3、四列に変わったら4に変えるだけだ」

```perl
package Filter::TransformFields;
use Moo;
sub process {
    my ($self, $rows) = @_;
    return [ map {
        my ($name, $amount, $date) = @$_;
        $name   =~ s/^\s+|\s+$//g;
        $amount =~ s/[^0-9]//g;
        $date   =~ s/^\s+|\s+$//g;
        { name => $name, amount => int($amount), date => $date };
    } @$rows ];
}
```

「五人目、フィールド変換。トリムと数値化」

```perl
package Filter::ValidateAmount;
use Moo;
sub process {
    my ($self, $records) = @_;
    return [ grep { $_->{amount} > 0 } @$records ];
}
```

「六人目、金額バリデーション」

```perl
package Filter::CalculateTotal;
use Moo;
sub process {
    my ($self, $records) = @_;
    my $total = 0;
    $total += $_->{amount} for @$records;
    return { records => $records, total => $total };
}
```

「そして七人目、合計計算。全員が個別の取調室を持った」

ロックはパイプラインを組み立てた。

```perl
my $pipeline = CsvPipeline->new(filters => [
    Filter::SkipBlankLines->new,
    Filter::SkipHeader->new,
    Filter::ParseColumns->new,
    Filter::ValidateColumnCount->new,
    Filter::TransformFields->new,
    Filter::ValidateAmount->new,
    Filter::CalculateTotal->new,
]);

my $result = $pipeline->execute(\@csv_lines);
```

「見たまえ。七つのフィルターが一列に並んでいる。データはこのパイプラインを上から下へ流れ、各フィルターで一つずつ加工される」

私はコードを見つめた。Before では一つのメソッドに詰め込まれていた七つの処理が、七つの独立したクラスに分離されている。

「これなら……合計計算を直しても、バリデーションには触れませんね」

「その通り。そしてもう一つ。フィルターの順番を入れ替えたり、新しいフィルターを追加したり、不要なフィルターを外したりするのも、この配列を書き換えるだけだ」

「各フィルターは入力を受け取り、出力を返す。それだけだ。隣の房の人間が何をしているかは知らない。知る必要もない」

## 事件解決：それぞれの取調室

テストを走らせた。

```
# Subtest: After: 各フィルター単体テスト — SkipBlankLines
ok 1 - 空行が除去される

# Subtest: After: 各フィルター単体テスト — SkipHeader
ok 1 - ヘッダーがスキップされる

# Subtest: After: 各フィルター単体テスト — ValidateColumnCount
ok 1 - カラム数不正の行が除去される

# Subtest: After: パイプライン全体 — Before と同じ結果を返す
ok 1 - レコードが3件
ok 2 - 名前がトリムされている
ok 3 - 金額が数値化されている
ok 4 - 合計が正しい

# Subtest: After: フィルターの組み替えが容易
ok 1 - 合計計算なしのためARRAYが返る
ok 2 - レコードが3件
```

全テスト、警告ゼロでパスした。各フィルターが個別にテストでき、パイプライン全体も Before と同じ結果を返す。

「フィルターを一つ追加するだけで、他の処理に一切影響しない……！」

「全員同房から個別の取調室に移しただけだよ。一人を取り調べても、隣の房には響かない」

私は来月追加予定の「税率計算フィルター」のことを考えた。Before なら、あの巨大メソッドのどこかに十行ほどねじ込む作業だった。After なら、`Filter::ApplyTax` を一つ書いて、パイプラインの配列に追加するだけだ。

「報酬は、このパイプラインのフィルター数と同じ杯数のエスプレッソでいい」

七つのフィルターで七杯。

「……七杯も飲んだら眠れなくなりませんか」

「灰色の脳細胞を活性化させるにはカフェインが必要なんだよ、ワトソン君」（それはポワロのセリフだと思うけど、指摘するのはやめておいた）

## 探偵の調査報告書

| 容疑（アンチパターン） | 真実（パターン） | 証拠（効果） |
|---|---|---|
| God Method — 空行除去、ヘッダースキップ、カラム分割、バリデーション、変換、合計計算が一つのメソッドに詰め込まれている。一箇所を修正すると別の処理が壊れる | Pipeline（Pipes and Filters） — 各処理ステップを独立したフィルタークラスに分離し、パイプラインがデータを順番に流す | `import_csv` の七つの責務が七つのフィルターに分離され、各フィルターを個別にテスト・交換・追加できるようになった |
| テスト困難 — 個別の処理ステップを単体テストする手段がない。空行除去だけをテストするにもメソッド全体を呼ぶ必要がある | 単体テスト可能 — 各フィルターが `process` メソッドを持つ独立したクラスのため、個別にテスト可能 | `Filter::SkipBlankLines` 単体で空行除去をテスト、`Filter::ValidateColumnCount` 単体でカラム数検証をテストできる |

### 推理のステップ

1. **God Method の責務を数える** — メソッド内のコメントや処理ブロックを手がかりに、独立した責務（空行除去、ヘッダースキップ、カラム分割、バリデーション、変換、合計計算など）を洗い出す
2. **各責務をフィルタークラスに切り出す** — 一つの責務につき一つの `Filter::*` クラスを作り、`process` メソッドで入力を受け取り出力を返すようにする
3. **パイプライン本体を実装する** — `CsvPipeline` のように、フィルターの配列を受け取り、`execute` でデータを順番に流すクラスを作る
4. **各フィルターの単体テストを書く** — 各フィルターに個別のテストデータを渡し、期待する出力を検証する。God Method では不可能だった単体テストが可能になる
5. **パイプラインの統合テストで Before と同じ結果を確認する** — 全フィルターを組み立てたパイプラインが、元の God Method と同じ結果を返すことを検証する

### ロックより

一つのメソッドにすべてを詰め込むのは、容疑者を全員同じ房に閉じ込めるようなものだ。一見すると管理が楽に見える。だが実際には、一人が動くたびに全員が影響を受ける。修正のたびに壊れるメソッドは、過密な独房の産物だ。

Pipeline パターンは、各処理に個別の部屋を与える。各フィルターは自分の仕事だけを知り、入力を受け取り、出力を返す。隣のフィルターの存在すら知らない。この無関心こそが、変更に強いコードの秘訣だ。

容疑者を全員同房にするな。一人ずつ、個別の取調室に入れたまえ、ワトソン君。
